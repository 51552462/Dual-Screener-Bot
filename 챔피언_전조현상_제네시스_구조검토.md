# 챔피언 전조현상(Genesis Precursor) 자율 축적·검증 구조 검토

> 목적: **수익률 높은 챔피언 로직이 탄생하면, 그 로직이 돈을 벌기 시작한 시점의 "한 달 전 환경"을
> 우리 시스템 내부 데이터로만(외부 금지·필터링) 포착·축적**하고, 그 전조가 **실제로 챔피언의 진짜 수익과
> 인과적으로 이어지는지(적중률)** 를 자율 검증하는 구조 설계.
> 본 문서는 **구조 검토(설계) 전용** — 코드는 수정하지 않음. 모든 파일/테이블/컬럼명은 실제 코드 기준.

---

## 0. 개념 정의 — "전조현상(Genesis Precursor)"이란

```
[T-30: 전조 윈도]            [T0: 점화 Ignition]         [T+: 챔피언 등극]
우리 구조 환경 스냅샷    →    챔피언 로직이 알파를     →    데스매치 우승
(국면벡터·수급·순환매·            뿜기 시작한 날                (deathmatch_champion)
 PRI·breadth·스카웃 흔적)
        │                                                        │
        └──────────── 이 둘이 통계적으로 연결되는가? (적중률 검증) ──┘
```

- **전조현상** = 챔피언이 돈을 벌기 시작한 점화일(T0) **직전 약 30일** 동안 우리 시스템이 이미 관측·저장한
  내부 환경 신호의 묶음(벡터).
- **핵심 제약(필터링)**: 전조는 **반드시 "우리 구조와 환경 안에 있는" 데이터**로만 구성한다.
  외부 뉴스/매크로 원시 피드를 즉석에서 끌어오지 않고, **이미 우리 SQLite/config_kv에 박제된 내부 스냅샷**만 사용.
- **검증 목표**: "이 전조가 떴을 때 → 한 달 뒤 진짜 챔피언이 태어나고 → 진짜 수익이 났는가"의 **적중률(Hit Rate)**.

## 1. 현재 구조에서 챔피언은 어떻게 태어나는가 (출발점)

| 단계 | 모듈 / 테이블 | 산출물 |
|---|---|---|
| 로직 식별 | `evolution/deathmatch_battle_royale.ledger_group_key` | `group_key`(선취매는 `_PREBUY/_SPILLOVER`로 이미 분리됨) |
| 스코어링 | `evolution/deathmatch_scorecard.compute_composite_v2` | `composite_score`, `hurdle_passed`, `champion_eligible` |
| 챔피언 선정 | `run_nway_deathmatch` (`:346-359`) | `champion_eligible & rank 최소` → `out.champion` |
| 영속화 | `evolution/deathmatch_store.deathmatch_champion` | `champion_label`, `composite_score`, `mean_ret`, **`as_of_date`**, `run_id` |
| 일별 추적 | `deathmatch_arm_snapshot` (trade_date별) | arm별 시계열(점화 시점 역추적 가능) |

**중요**: 챔피언은 **`group_key`** 단위이고, 그 거래 이력은 `forward_trades`에 `sig_type`(→group_key)·`entry_date`로 남아 있다.
즉 "이 챔피언이 언제부터 돈을 벌었나(T0)"는 `deathmatch_arm_snapshot`/`forward_trades`로 역추적 가능하다.

## 2. 우리 구조 내부 '전조 데이터' 인벤토리 (필터링 통과 소스만)

전조 벡터는 아래 **이미 저장된 내부 신호**만으로 구성한다(외부 즉석 호출 금지 = 필터링 규칙 충족).

| 소스 | 위치 | 전조 신호로서의 의미 | 시간 인덱스 |
|---|---|---|---|
| **국면 벡터 롤링버퍼** | `REGIME_VECTOR_HISTORY` (config_kv, 180일) | spx/kospi MA20이격·VIX_z·**PRI_z**·macro_z·range5d — "환경의 모양" | ts(일별) ✅ |
| 국면 유사도 | `REGIME_ANALOG_SCORE` (best_episode, score) | 당시가 어떤 역사적 국면과 닮았나 | updated_at |
| 5-Factor 앙상블 | `REGIME_ENSEMBLE` / `CURRENT_REGIME_KEY` | 당시 확정 국면·확신도(softmax) | 일별 |
| **섹터 순환매** | `sector_daily_leader` / `sector_transition_rollup` | 자금이 챔피언 섹터로 **이동 중이었나**(전이 궤적) | trade_date ✅ |
| 다음 섹터 예측 | `PREDICTED_NEXT_SECTOR_*` / `US_SPILLOVER_SECTOR` | 챔피언 섹터를 미리 가리켰나 | as_of ✅ |
| **스마트머니** | `SMART_MONEY_RADAR(_US)` | 외인·기관/다크풀이 챔피언 종목군에 들어와 있었나 | updated_at ✅ |
| 시장 폭 | `forward_trades.entry_breadth` / `market_breadth` | 당시 시장 폭(매집 환경) | entry_date ✅ |
| **로직 조기 흔적** | `forward_trades` (`OBSERVE_ONLY`/`SCOUT`/`INCUBATOR_` 태그) | 챔피언 group_key가 **관측/스카웃 단계**로 이미 꿈틀댔나 | entry_date ✅ |
| 진입 DNA | `forward_trades.dyn_cpv/dyn_tb/v_energy/v_rs/total_score` | 챔피언 거래의 수급·에너지 지문 | entry_date ✅ |
| 밴딧 사전확률 | `template_bandit` Beta(α,β) | 등극 직전 승률 사전분포가 끓고 있었나 | 갱신시각 |

→ 전조 벡터는 **"환경(국면/순환매/수급/breadth) + 로직 자신의 태동(스카웃/인큐베이터/DNA)"** 두 축으로 구성한다.
모두 우리 DB/config 안의 값 → **필터링(우리 구조 한정) 자동 충족**.

## 3. 제안 구조 — Champion Genesis Precursor Engine

신규 모듈 제안: `evolution/champion_genesis.py` (기존 데스매치/앙상블 **불간섭**, 읽기 중심 + 전용 테이블 기록).

### 3.1 트리거 & 점화일(T0) 정의

- **트리거**: 데스매치 1사이클 종료 후 `deathmatch_champion`이 갱신될 때(또는 직후 별도 배치)에 한 번 호출.
- **점화일 T0 산정**(역추적, 셋 중 채택):
  1. `deathmatch_arm_snapshot`에서 해당 group_key의 `composite_score`(또는 누적 mean_ret)가 **처음 양(+)의 우위로 전환된 trade_date**.
  2. 그게 모호하면 `forward_trades`에서 챔피언 group_key의 **누적 수익곡선이 신고점을 찍기 시작한 날**.
  3. 최후 폴백: 챔피언 첫 `CLOSED_WIN` 군집의 entry_date 중앙값.
- **전조 윈도** = `[T0 - 30영업일, T0]`.

### 3.2 전조 벡터 스키마 (윈도 집계)

```jsonc
// champion_precursor_genesis.precursor_json (예시)
{
  "env": {                          // 환경 축 (REGIME_VECTOR_HISTORY 윈도 평균/기울기)
    "regime_vec_mean": [..6dim..],  // 윈도 평균 (spx,kospi,vix_z,pri_z,macro_z,range5d)
    "regime_vec_slope": [..6dim..], // 윈도 기울기(전조의 '방향성')
    "pri_z_trend": 0.42,            // 내부 PRI가 끓어오르고 있었나
    "analog_episode": "V_RECOVERY", // 당시 닮은 역사 국면
    "regime_key_path": ["SIDEWAYS","SIDEWAYS","BULL"], // 전이 경로
    "breadth_mean": 1.01
  },
  "rotation": {                     // 순환매 축
    "sector": "반도체/IT",
    "was_predicted_next": true,     // PREDICTED_NEXT_SECTOR가 미리 가리켰나
    "spillover_aligned": true,      // US_SPILLOVER_SECTOR 일치
    "dominance_gain_days": 6        // 챔피언 섹터 주도일 증가
  },
  "smart_money": {                  // 수급 축
    "radar_hits_in_sector": 3,      // 챔피언 섹터에 스마트머니 픽 존재
    "avg_divergence_score": 38.5
  },
  "genesis_footprint": {            // 로직 태동 축 (챔피언 자신)
    "scout_trades": 4,              // OBSERVE_ONLY/SCOUT 조기 표본
    "incubator_seen": true,
    "entry_dna_mean": {"cpv":0.71,"tb":12.4,"v_energy":28.0}
  }
}
```

- 환경 축은 `REGIME_VECTOR_HISTORY`의 **윈도 평균 + 기울기**로 "정적 상태 + 변화 방향"을 함께 담는다
  (regime_analog_engine의 6차원 VECTOR_DIMS 재사용 → 호환성·해석성 확보).
- 모든 값은 윈도 내부에서만 집계 → **미래누설(look-ahead) 차단**.

### 3.3 필터링 규칙 (우리 구조 내부만 — 핵심 제약)

1. **내부 SSOT만**: config_kv(REGIME_*·SMART_MONEY_*·PREDICTED_*·US_SPILLOVER_*) + market DB(`forward_trades`,`sector_*`,`deathmatch_*`). 즉석 yfinance/pykrx/뉴스 호출 **금지**.
2. **윈도 경계 엄수**: `entry_date/trade_date/ts ≤ T0` 만 포함(점화 이후 데이터로 전조를 오염시키지 않음).
3. **챔피언 자기 거래 제외 옵션**: 환경 축은 시장 전체, 태동 축만 챔피언 group_key로 한정해 **환경 vs 로직** 기여를 분리.
4. **품질 게이트**: 윈도 내 유효 표본(섹터 일자 ≥ N, 국면벡터 점 ≥ 10)일 때만 전조를 "유효"로 저장. 아니면 `status='thin'`.

### 3.4 축적 테이블 (신규, 데스매치 DB 동거)

```sql
CREATE TABLE IF NOT EXISTS champion_precursor_genesis (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    market TEXT NOT NULL,
    champion_label TEXT NOT NULL,      -- group_key (선취매면 _PREBUY/_SPILLOVER 포함)
    ignition_date TEXT NOT NULL,       -- T0
    crowned_date TEXT,                 -- 데스매치 등극일 (deathmatch_champion.as_of_date)
    window_start TEXT, window_end TEXT,
    precursor_json TEXT NOT NULL,      -- 3.2 스키마
    champion_composite REAL,           -- 등극 시 점수(라벨 강도)
    realized_fwd_ret REAL,             -- 등극 후 N일 실현수익(인과 검증용, 후속 갱신)
    status TEXT DEFAULT 'pending',     -- pending|confirmed|failed|thin
    UNIQUE(market, champion_label, ignition_date)
);
```

→ 이렇게 챔피언이 나올 때마다 한 줄씩 쌓이면 **"챔피언이 태어나기 직전 우리 세상의 모양" 라이브러리**가 된다.

## 4. 검증 루프 — 전조 → 챔피언 → 실수익 적중률

전조를 **쌓기만** 하면 미신이 된다. 진짜 인과를 보려면 두 방향 검증이 필요하다.

### 4.1 (사후) 인과 검증 — "정말 챔피언 수익으로 이어졌나"

- 챔피언 등극 후 `realized_fwd_ret` = 등극일~+N일 챔피언 group_key의 실현 PnL(`forward_trades`)을 후속 채움.
- `realized_fwd_ret > 0` & 일정 기준 → `status='confirmed'`, 아니면 `failed`.
- → "전조가 떴고 → 챔피언이 됐는데 → 실제로 돈을 벌었나"를 라벨링(전조 라이브러리의 진위 검증).

### 4.2 (선행) 예측 검증 — "현재 전조가 미래 챔피언을 부르는가"

- 매일/주간, **현재 환경 벡터**(REGIME_VECTOR_HISTORY 최근 윈도)를 축적된 `confirmed` 전조들과 **유사도 비교**.
  - 재활용: `regime_analog_engine`의 **마할라노비스 + DTW 앙상블**(이미 구현됨) → 별도 수학 불필요.
  - 유사도 ≥ 임계(예: 0.80) & 해당 전조의 섹터/아키타입 → "챔피언 인큐베이션 경보" 발령(섹터 X에서 한 달 내 챔피언 가능성).
- **적중률 원장**(신규):

```sql
CREATE TABLE IF NOT EXISTS precursor_prediction_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    predict_date TEXT, market TEXT,
    matched_champion_label TEXT,    -- 닮은 과거 전조의 챔피언
    matched_sector TEXT,
    similarity REAL,
    horizon_days INTEGER DEFAULT 30,
    -- 후속 채움(검증):
    hit INTEGER,                    -- 30일 내 해당 섹터/아키타입 챔피언 실제 등극?
    realized_ret REAL,              -- 그 챔피언이 실제로 번 수익
    recorded_at TEXT
);
```

- `hit`/`realized_ret`는 horizon(30일) 경과 후 자동 채움 → **EMA 적중률·평균 초과수익** 산출.
- 이 적중률이 곧 "전조 구조의 가치"를 자가 증명하는 지표. (낮으면 전조 가중·임계 자동 강등 = 자율 진화)

### 4.3 데스매치/켈리와의 연결(선택, 자율 진화)

- `confirmed` 전조와 현재가 고유사 + 적중률이 검증된 경우에 한해, 해당 섹터/아키타입의 **신규 스카웃 진입 켈리를 소폭 가산**
  (기존 `ROTATION_ADVANTAGE_ACTIVE`/`SPILLOVER_ADVANTAGE_ACTIVE`와 동일 사상: **증명된 것만 베팅↑**).
- 단 이는 4.2 적중률이 통계적으로 유의할 때만 잠금 해제(돈 벌 때만 베팅).

## 5. 기존 구조 비훼손 통합 포인트 (방어적 훅)

| 위치 | 훅 방식 | 비훼손 보장 |
|---|---|---|
| 데스매치 종료 후 | `save_battle_royal_result` **호출자 측**에서 `champion_genesis.capture(...)` try/except 호출 | 데스매치 본체 무수정, 실패해도 데스매치 영향 0 |
| 점화일 역추적 | `deathmatch_arm_snapshot` / `forward_trades` **읽기 전용**(RO URI) | 원장 쓰기 없음 |
| 전조 벡터 | `REGIME_VECTOR_HISTORY` 등 **config 읽기만** | 앙상블/게이트 무영향 |
| 적중률 검증 | 신규 테이블 2개에만 기록 | 기존 스키마 불변 |
| 켈리 연동(선택) | 신규 플래그 `GENESIS_PRECURSOR_ADVANTAGE`로 게이트 | 기본 OFF, 증명 시만 ON |

원칙: **신규 모듈 + 신규 테이블 + 읽기 전용 소비**. 5-Factor 앙상블·데스매치 랭킹·NAV 산출 경로는 일절 건드리지 않는다.

## 7. 단계별 구현 로드맵 (제안)

1. **P1 축적기**: `evolution/champion_genesis.py` + 테이블 2개 DDL. 데스매치 호출자에 try/except 훅. (전조 라이브러리 쌓기 시작)
2. **P2 점화일 역추적**: `deathmatch_arm_snapshot` 기반 T0 산정 함수 + 전조 윈도 집계.
3. **P3 사후 인과 검증**: `realized_fwd_ret` 후속 배치 → confirmed/failed 라벨.
4. **P4 선행 예측 검증**: 현재 벡터 vs confirmed 전조 유사도(regime_analog 재활용) → `precursor_prediction_log` + 적중률 EMA.
5. **P5 리포트**: 텔레그램/주간 리포트에 "챔피언 인큐베이션 경보 + 전조 적중률" 섹션(읽기 전용).
6. **P6(선택) 자율 베팅 연동**: 적중률 유의 시 `GENESIS_PRECURSOR_ADVANTAGE` 잠금 해제.

## 9. 한 줄 결론

우리 구조엔 이미 **챔피언 결정 경로(`deathmatch_champion`)** 와 **한 달 전 환경을 그대로 담은 내부 시계열
(`REGIME_VECTOR_HISTORY` 180일·`sector_*`·`SMART_MONEY_RADAR`·`forward_trades` 조기흔적)** 이 다 있다.
빠진 것은 이 둘을 잇는 **(1) 점화일 역추적 → 전조 벡터 축적기, (2) 전조→챔피언→실수익 적중률 검증 루프** 두 조각뿐이다.
이를 **신규 모듈 + 신규 테이블 + 읽기 전용 소비**로 붙이면, 기존 5-Factor 앙상블·데스매치를 전혀 훼손하지 않고
"우리만의 전조현상 라이브러리"를 자율로 쌓고 **돈으로 증명된 전조만 베팅에 반영**하는 폐루프를 완성할 수 있다.

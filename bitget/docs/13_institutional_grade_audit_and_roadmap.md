# Bitget 퀀트 구조 — "1년 방치 무인 복리" & "기관/헤지펀드급" 종합 감사 보고서

**작성일:** 2026-07-04
**목적:** 이 문서는 Cursor(Claude)가 `bitget/` 코드베이스 전체를 정적 분석한 결과를 정리한 것으로, Gemini와 함께 다음 단계 구현을 계획·분담하기 위한 SSOT(Single Source of Truth) 자료입니다.
**질문:** "1개월/3개월/1년을 그대로 방치해도 서버가 안 터지고, 저절로 복리가 매달 우상향할 수 있는 구조인가? 기관·헤지펀드를 뛰어넘는 개인 퀀트가 될 수 있는가?"
**결론(한 줄 요약):** **아니오, 아직은 아닙니다.** 서버 자체가 몇 달간 안 죽는 것은 상당 부분 가능하지만(로그/디스크 이슈 제외), "복리가 저절로 우상향"하려면 자본배분·리스크관리·전략진화 세 축에서 구조적 결함을 먼저 고쳐야 합니다. 현재 시스템은 **아직 실계좌 매매가 아니라 가상매매(paper trading) 기반**이라는 점이 가장 중요한 전제입니다.

---

## 0. 가장 중요한 전제: 지금 이 시스템은 "실전 매매"가 아니다

`ENABLE_REAL_EXECUTION=false`, `REAL_EXECUTION_DRY_RUN=true`가 기본값이며, 실제 손익의 기록부(book of record)는 거래소 계좌가 아니라 SQLite `bitget_forward_trades` 테이블입니다. 즉:

- 지금 매일 보는 "복리 우상향" 리포트는 **가상 시뮬레이션 PnL**입니다.
- 슬리피지 미반영, 펀딩비 미차감, 실제 주문 체결가 미반영 등으로 인해 **가상 PnL은 실제보다 낙관적으로 나올 가능성이 높습니다.**
- 실전 전환(`ENABLE_REAL_EXECUTION=true`)을 언제, 어떤 조건으로 할지가 이 로드맵의 최종 목적지여야 합니다.

이 문서의 모든 개선 항목은 "가상매매의 논리적 완성도"와 "실전 전환 시 필요한 안전장치"를 함께 다룹니다.

## 1. 카테고리별 현황 요약

### 1.1 리스크·자본관리 (Kelly, Treasury, Drawdown, Leverage)

| 항목 | 존재 여부 | 자동/수동 | 평가 |
|---|---|---|---|
| Kelly 사이징 | 있음 (fractional risk %, 고전적 Kelly 공식 아님) | 자동, 다층 캡 | 준수하나 하드캡(25%)이 감사(audit)에만 있고 실매매 경로엔 강제 안 됨 |
| Treasury(자본풀) | 있음, SPOT/FUTURES 분리 | 자동 회계 | `ACCOUNT_SIZE_USDT`가 정적 앵커라 그룹 단위로만 부분 복리 |
| Tail-risk fund | 있음 | 자동 적립이지만 **매매 로직에서 전혀 안 읽힘** | 사실상 장식용 회계, 진짜 서킷브레이커 아님 |
| Drawdown 서킷브레이커 | 있음 (오픈손실 -5%, 최근 20건 -2000USDT, 그룹별 MDD -30%) | 자동 | **계좌 전체(NAV) 레벨의 -15%/-20%/-30% 셧다운은 없음** — 전부 부분적(그룹/샘플 단위) |
| 레버리지 상한 | 기본 3x, **하드 MAX_LEVERAGE 없음** | 자동(사이즈 계산만) | 청산가 실시간 모니터링 없음 — 단순 ROE 근사 모델 |
| 상관관계/집중 리스크 | **없음** | — | 포지션당 %캡 + 최대 20개 오픈 수만 존재. "다 같이 떨어지는" 알트 대량 하락 시나리오 무방비 |
| 펀딩비 리스크 | 청산 시점 bleed-stop만 | 자동(청산만) | 진입 시점엔 필터 없음, PnL 계산에도 미반영 |
| KILL_SWITCH | 있음 | **완전 수동** | 이상 감지로 자동 발동되는 코드 없음 |

**핵심 결함:** 계좌 전체 단위의 최대 손실 한도(portfolio-level max drawdown breaker)가 없다. 그룹 단위 -30% MDD는 있지만, 여러 그룹이 동시에 -25%씩 손실 나면 계좌는 -25%가 나도 아무 그룹도 -30% 기준을 넘지 않아 전체 서킷브레이커가 발동하지 않을 수 있음.

### 1.2 서버 복원력 (Watchdog, Lock, DB, 디스크, 재부팅)

| 항목 | 상태 | 평가 |
|---|---|---|
| Watchdog | heartbeat 기반, **factory만 자동 재시작** | WS/queue-worker는 알림만 감(사람이 수동 재시작해야 함) |
| WebSocket 재연결 | 지수 백오프 + systemd 재시작 | 양호 |
| Lock 자동복구 | PID/나이 기반 자동 해제 | 양호 |
| DB 백업 | 5분마다 CQRS 스냅샷 | **PRAGMA integrity_check 없음**, 무결성 검증된 백업(`institutional_db_backup.py`)이 cron에 연결 안 됨 |
| 로그/디스크 | **로테이션 없음** | 27개 스캔 슬롯 × 매 실행마다 새 타임스탬프 로그 파일 생성 → **수개월 방치 시 디스크 고갈 확실** |
| 메모리 누수 | 감지 없음, MemoryMax OOM-kill만 | 서서히 새는 메모리는 장기간 미탐지 가능 |
| API 레이트리밋 | 호출 단위 백오프 양호 | 거래소 전체 다운 시 서킷브레이커 없음 |
| 크론 오버랩 | flock + 최대 50분 대기 + 스킵 | 스캔이 50분 넘으면 그 슬롯은 조용히 스킵(커버리지 저하, 크래시는 아님) |
| 재부팅 복구 | systemd enable + cron.d | 양호하나 watchdog 타이머가 `Persistent=true` 아님 |

**핵심 결함:** **로그 파일 무제한 증가가 "1년 방치"의 가장 확실한 서버 파괴 요인**이다. 나머지는 대부분 이미 잘 되어 있음.

### 1.3 전략 진화 (Deathmatch, 수명주기, 과최적화 방지)

| 항목 | 상태 | 평가 |
|---|---|---|
| Deathmatch 랭킹 | 매일 자동 실행 | **Bitget은 `apply_deathmatch_allocation=False`로 명시적으로 자본배분을 꺼둠** — 랭킹은 정보성일 뿐, 나쁜 전략이 자동으로 굶지 않음 |
| 시장 키 불일치 | `market="BG"` vs `market="SPOT"/"FUT"` | Deathmatch와 MetaGovernor 레지스트리가 서로 다른 키로 나뉘어 있어 정합성 약함 |
| Champion genesis(전조 학습) | 코드는 있으나 **Bitget 일일 리포트 경로에 연결 안 됨** | "무엇이 통했는지" 전조 라이브러리가 축적되지 않음 |
| 전략 레지스트리 수명주기 | OBSERVING→CANDIDATE→LIVE→COOLED→RETIRED 자동 전환 | 양호 |
| 과최적화 방지 | 최소 표본수 게이트 존재, 14일 OOS(engine6 한정) | **Bitget 파이프라인 전체에 걸친 walk-forward 검증은 없음** — 90일 롤링 윈도우가 사실상 in-sample |
| 시그널 유니버스 | 코어 5개 + 실무자룰 30개 + 인큐베이터 뮤턴트 | **고정된 코드 기반** — 새 "규칙 종류" 추가는 코드 수정 필요, 파라미터 변이만 자동 |
| 레짐 적응 | Kelly/가중치엔 영향 O | **어떤 엔진을 돌릴지는 레짐과 무관하게 고정 크론** |
| 코인 유니버스 | 거래량 기준 자동 갱신 | 양호 |
| 버그 | `master_scanner.py:97` `bse` NameError (전체 스캔 시) | staggered cron이라 현재는 회피되지만 잠재 버그 |

**핵심 결함:** 나쁜 전략을 실제로 굶기는 메커니즘(deathmatch allocation)이 Bitget에서 꺼져 있다. 레지스트리 수명주기가 있다 해도, deathmatch의 "상대평가"가 자본배분에 반영 안 되면 시장이 바뀔 때 죽은 전략에 계속 자본이 갈 수 있음.

### 1.4 주문 실행 품질 (실전 전환 시 필요)

| 항목 | 상태 |
|---|---|
| 실전 매매 | **기본 꺼짐** (paper/dry-run) |
| 실전 진입 | market order만, 청산(exit)은 **실주문 코드 자체가 없음** |
| 슬리피지 | 가상 PnL엔 전혀 반영 안 됨(낙관적 편향) |
| 스마트 실행 | TWAP/iceberg 등 전혀 없음 — 계좌 커지면 시장충격 문제 |
| 청산가 모니터링 | 실제 거래소 청산가 아닌 단순 ROE 근사(-100%/레버리지) |
| 펀딩비 PnL 반영 | 추적만 하고 실현손익 계산에서 차감 안 함 |
| 실주문 사이즈 | 가상 사이즈와 분리, 실패 시 20 USDT 폴백 — **가상/실전 사이즈 불일치** |
| 총 노출 한도 | 포지션당 %캡만 있고 **포트폴리오 전체 명목가 상한 없음** |
| 고아 포지션 처리 | 알림만, 자동 정리 없음 |

**핵심 결함:** 만약 지금 상태로 실전 전환 스위치를 켠다면 (1) 청산 주문이 안 나가고 (2) 실제 사이즈가 가상 사이즈와 안 맞고 (3) 펀딩비가 실현손익에서 안 빠지는 세 가지 때문에 큰 사고가 날 수 있음. **지금은 절대 실전 전환하면 안 되는 상태.**

### 1.5 관측/알림/휴먼 개입

| 항목 | 상태 |
|---|---|
| 이상 감지(가격 급변/배드틱) | **없음** — 이상 캔들이 그대로 시그널/손절 로직에 들어감 |
| 텔레그램 알림 | 과다(신호마다 알림) + 과소(성공 실행은 조용, 예상된 스킵도 조용) 양쪽 리스크 공존 |
| 메타 상태 오염 방지 | scan/daily 진입 전 체크 후 중단(abort) — 양호 |
| Config 값 범위 검증 | **쓰기 시점 검증 없음** — Kelly mult, cutoff 등이 이론상 무한히 표류 가능(감사는 사후 확인만) |
| 휴먼 승인 게이트 | **없음** — 주간 액션플랜도 자동으로 baseline을 저장하고 적용, 사람 승인 절차 없음 |
| 비용 모니터링 | **없음** — Gemini API 비용, 거래소 수수료 추적 안 됨 |
| 실전 vs 가상 괴리 감지 | baseline fingerprint 비교만 있고 진짜 "실행 성과 vs 시뮬레이션 성과" 비교 없음 |

## 2. "기관/헤지펀드급"이 되려면 추가로 필요한 것

기관 퀀트펀드와 개인 알고리즘 트레이딩 시스템의 근본적 차이는 다음 5가지입니다. 현재 구조는 3, 4, 5번이 특히 약합니다.

1. **포트폴리오 레벨 리스크 관리** (VaR, 상관관계 행렬, 팩터 노출) — 현재는 전략 단위 리스크만 있고 계좌 전체를 하나의 포트폴리오로 보는 레이어가 없음.
2. **엄밀한 워크포워드/OOS 검증** — 현재는 하드코딩된 임계값 + 부분적 OOS만 있어, 과최적화 위험이 상존.
3. **실행 알고리즘(execution algo)** — 시장충격을 최소화하는 스마트 주문 실행이 전무. 계좌가 커질수록 이 부분 없이는 수익률이 깨짐.
4. **자본 스케일링 로직** — 계좌가 10배, 100배 커져도 같은 전략이 같은 초과수익을 낼 수 없음(유동성 한계). 현재 시스템은 이를 전혀 모델링하지 않음.
5. **거버넌스/감사추적** — 기관은 모든 리스크 파라미터 변경에 승인 절차와 불변 로그가 있음. 현재는 완전 자동+자가수정 구조라 "왜 이렇게 됐는지" 사후 추적이 약함.

## 4. 결론

현재 구조는 **"몇 달간 서버가 죽지 않고 계속 도는 것"** 자체는 로그 로테이션 하나만 고치면 상당히 안정적입니다 (watchdog, lock 자가치유, WS 재연결, DB 스냅샷 등은 이미 잘 구현되어 있음).

그러나 **"저절로 복리가 우상향"** 하려면:
1. 나쁜 전략이 실제로 도태되어야 하는데, 지금은 deathmatch 자본배분이 꺼져 있어 이 메커니즘이 작동하지 않고,
2. 전체 계좌 단위의 안전장치(NAV 드로다운 서킷브레이커, 레버리지 캡, 상관관계 리스크)가 없어 특정 시나리오(여러 그룹 동시 손실, 알트코인 동반 급락, 과도한 레버리지)에서 큰 손실이 날 수 있으며,
3. 지금은 **실전 매매가 아니라 가상매매**이므로, "복리"가 실제 계좌에 반영되려면 실전 전환 전에 실행 품질(청산 주문, 사이즈 정합성, 펀딩비 반영, 슬리피지)을 먼저 완성해야 합니다.

**기관/헤지펀드를 뛰어넘으려면** 위 P2 항목들(워크포워드 검증, 자본 스케일링, 실행 알고리즘, 거버넌스)이 핵심이며, 이는 대부분 "더 정교한 리스크 관리 인프라"의 문제이지 "더 나은 신호(alpha)"의 문제가 아닙니다. 개인 퀀트가 기관을 이길 수 있는 지점은 보통 여기(리스크관리 정교함 + 실행 품질)이지, 신호 발굴 경쟁에서가 아닙니다.

## 5. Gemini와의 협업을 위한 제안

이 문서를 기반으로 Gemini에게는 다음을 요청하는 것을 제안합니다:
- P0/P1 항목 중 코드 구현이 필요한 부분의 구체적 설계안 (특히 P0-2 NAV 서킷브레이커, P1-1 deathmatch 자본배분 재활성화, P1-2 상관관계 캡)
- P2 항목 중 워크포워드 검증 프레임워크의 아키텍처 설계 (기존 `validation/walk_forward.py`를 Bitget에 어떻게 통합할지)
- 이 로드맵에 대한 반론이나 놓친 리스크 요인이 있는지 크로스체크

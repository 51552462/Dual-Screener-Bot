## 1. [#3] DB Writer 직렬화 & Busy Timeout 통일 — 청산 실패 방지

### 1.1 무엇을 했나 (적용=ON, 순수 추가)

**(A) 공용 유틸 강화 — `low_ram_sqlite_pragmas.py`**
- `apply_busy_timeout(conn, ms=60000)` 신설.
- `apply_oom_safe_pragmas()`에 `busy_timeout=60000` 포함 → 이 헬퍼를 이미 호출하던 경로(`market_data_fetcher`)는 **자동으로** 락 내성 획득.
- `apply_lock_safe_pragmas(conn, wal=...)` 신설(주식측 raw 연결 공용).
- 정책 상수(60000ms)는 코인 SSOT(`shared_db_connector.BUSY_TIMEOUT_MS`)와 **동일하게 통일**.

**(B) 주식 가상매매 장부 — `forward/shared.py`** (가장 중요한 청산/등재 write 경로)
- `init_forward_db`, 진입(`try_add_*`), 일일 마감·집계 등 **모든 연결에 `apply_busy_timeout` 적용**(읽기 전용 RO 연결 포함).

**(C) 주식 OHLCV 벌크 writer — `data_updater.py`**
- `timeout=15/30`이던 **짧은 연결 8곳**에 `busy_timeout=60000` 적용(스냅샷 백업 src/dst 포함).

**(D) 주식 시세 조회 — `market_data_fetcher.py`**
- `_fetch_sqlite` 연결 `timeout 30→60`s, 헬퍼로 busy_timeout 자동.

### 1.2 "글로벌 공용 write 락" 대신 무엇을 택했나 — 그리고 왜

요청은 "쓰기에 글로벌 공용 락(또는 직렬화 큐)"였다. 두 후보를 비교:

| 방식 | 장점 | 위험 |
|---|---|---|
| 별도 파일 락(flock)으로 모든 write 감싸기 | 강한 직렬화 | **교착(deadlock) 신규 위험** — 락 순서/예외 누수/24×7 데몬과 얽힘. 끊기면 전체 write 마비. |
| **busy_timeout=60000 + 단일 직렬 큐(#2)** | 추가 락 없이 SQLite 자체 대기열로 질서화. 큐 워커가 곧 직렬화 그 자체. | 거의 없음(추가 컴포넌트 0). |

→ **후자 채택.** WAL + `busy_timeout=60000`이면 동시 writer가 즉시 `database is locked`로 죽지 않고 **최대 60초 질서 있게 대기**한다. 청산(`track_positions`)의 짧은 트랜잭션은 이 대기 안에서 대부분 성공한다. 진정한 "직렬화"가 필요하면 그것은 **#2의 단일 큐 워커**가 담당한다(별도 락 도입 없이).

> 참고: Python `sqlite3.connect(timeout=N)`은 그 자체로 busy_timeout을 N초로 설정한다. 그래서 `timeout=60` 사이트는 이미 60초 대기가 있었지만, ① `data_updater`의 `timeout=15/30`처럼 **짧은 값**과 ② 일부 모듈이 `PRAGMA busy_timeout=7000`으로 **낮게 덮어쓰던 사고**(6/28 문서 지적)를 60000으로 통일해 못박은 것이 핵심 이득이다.

## 2. [#2] 코인 스캔 skip(Drop) 방지 → 대기(Queue) 전환

### 2.1 무엇을 했나 (코드 구현 완료, 활성화는 플래그 OFF 기본)

핵심 문제: 코인 스캔이 주식 factory에 **양보(yield)** 하면 그 회차가 **통째로 증발**(진입 데이터 누락).

기존 큐(`task_orchestrator`)는 실패 시 `fail()`로 재시도하지만 **attempts를 소모**해 `max_attempts(3)` 초과 시 영구 폐기된다. 주식이 몇 시간 락을 쥐면 코인 대기 작업이 3회 만에 죽어버린다. 그래서 **"실패가 아니라 대기"** 원시함수를 새로 넣었다.

**구현 (`bitget/infra/task_orchestrator.py`)**
- `TaskDeferred(available_in_sec=...)` 예외 신설.
- `enqueue(..., available_in_sec=0.0)` — 지연 적재 지원.
- `defer(task_id, available_in_sec, ...)` — **attempts를 소모하지 않고**(claim 시 +1된 것을 −1로 중립화) `available_in_sec` 뒤 PENDING 복귀.
- `process_one`이 `TaskDeferred`를 잡아 `defer()` 호출(→ 상태 `DEFERRED`).

**연결**
- `bitget/pipelines/queue_worker.py`: 워커가 yield 스킵(`skipped_session` + `"yield_to_factory"`)을 감지하면 `TaskDeferred` 발생 → **Drop 대신 60초 뒤 재시도**. 주식 락이 풀릴 때까지 무한정 대기 후 순차 실행.
- `bitget/infra/runtime.py` `dispatch_bitget_mode`: 인라인 cron 경로에서 yield 스킵 시 큐 적재(워커가 나중에 픽업). 워커 경로에서 호출되면 dedupe(현재 RUNNING)로 **무해하게 no-op**.

**게이트:** 모든 동작은 `BITGET_ENQUEUE_ON_YIELD=1`일 때만. 기본 OFF → **기존 동작 100% 보존**.

### 2.2 활성화 절차 (운영자가 켤 때)
1. 큐 워커 데몬 상시 가동: `python -m bitget.pipelines.queue_worker`(systemd 권장). 이미 `run_bitget_queue_worker.sh` 엔트리 존재.
2. (선택) cron을 `--enqueue` 모드로 전환 — `bitget` crontab 생성기에 큐 모드가 이미 있음.
3. `BITGET_ENQUEUE_ON_YIELD=1` 설정 → 재시작.
4. `task_orchestrator.backlog_stats()`로 PENDING/oldest_pending_age 모니터.

### 2.3 검증 (로컬 단위 테스트 통과)
- busy_timeout=60000 적용 확인.
- `defer`가 attempts를 0으로 중립화하고 미래 시점 PENDING으로 되돌림 + 즉시 재픽업 안 됨 확인.
- `enqueue(available_in_sec=2)`가 2초 후에만 픽업됨 확인.
- dedupe(PENDING/RUNNING 중복 차단) 확인.

## 3. [#1] 주식 글로벌 락 연쇄 지각 해소 (느린 스캔)

### 3.1 진단 재확인 — "로컬 우선"의 함정

요청은 "로컬 SQLite 최우선(Local First)"이었으나, **단순 재정렬은 장중 스캔 의미를 망가뜨린다:**
- KR 스캐너는 `[일봉 전용]`으로 **마지막 일봉**에 신호를 판정한다. 장중(10:00~15:20)엔 FDR가 **당일 형성 중인 봉**을 준다.
- 로컬 SQLite는 08:00 갱신본이라 장중엔 **전일 종가까지만** 보유. 무턱대고 로컬을 먼저 읽으면 스캔이 **전일 데이터로 판정** → 당일 신호 누락 → "원래 종목 많이 나왔는데" 라던 그 종목들이 **오히려 사라진다.**

즉 "로컬 우선"을 글로벌 기본값으로 켜면 사용자의 목표(당일 포착)와 정면충돌한다.

### 3.2 그래서 무엇을 했나 — "장애일 때만" 로컬로 빠른 폴백 (적용=ON, 정상시 불변)

진짜 원인은 **느린 스캔 자체**다: FDR/yfinance가 429/timeout으로 흔들리면 종목마다 3회 재시도(+지수 백오프) × 2개 네트워크 핸들러를 다 소진한 뒤에야 로컬로 폴백한다. 2-동시 세마포어와 곱해져 2483종 스캔이 **57분**까지 늘어지고 글로벌 락을 60분 점유 → 뒤 슬롯(`ema5`) 증발.

**구현 — `market_data_fetcher.py` 업스트림 저하 서킷 브레이커:**
- 프로세스 전역으로 최근 **일시적 실패(429/timeout/connection)** 건수를 윈도(기본 60s) 안에서 카운트.
- 임계(기본 40건) 초과 시 **트립** → cooldown(기본 120s) 동안 **네트워크 핸들러를 즉시 빈 응답으로 단락** → 체인이 곧장 로컬 SQLite로 폴백.
- cooldown 후 **자동 half-open 복구**.

**효과:** 폭풍이 와도 스캔이 **빠르게 끝나고** 글로벌 락을 **짧게** 쥔다 → `ema5` 등 뒤 슬롯 증발 방지. 정상일 때는 절대 트립하지 않으므로 **네트워크 우선 동작 그대로**(빈 응답/상장폐지는 카운트 안 함 — 오탐 방지).

**트레이드오프(명시):** 트립된 그 슬롯은 로컬(장중엔 보통 전일 종가)로 판정될 수 있다. 그러나 현 상태(57분 폭풍 + 0건 + 스케줄 붕괴)보다 "빠르게 끝내고 다음 슬롯/2회차에서 신선 데이터로 복구"가 운영상 훨씬 안전하다.

**튜닝 환경변수**
| 변수 | 기본 | 의미 |
|---|---|---|
| `MARKET_DATA_UPSTREAM_BREAKER` | `1` | 끄려면 `0` |
| `MARKET_DATA_UPSTREAM_BREAKER_FAILS` | `40` | 윈도 내 트립 임계 건수 |
| `MARKET_DATA_UPSTREAM_BREAKER_WINDOW_SEC` | `60` | 카운트 윈도 |
| `MARKET_DATA_UPSTREAM_BREAKER_COOLDOWN_SEC` | `120` | 트립 지속(최소 5s) |

검증: 트립/단락/SQLite 비차단/자동복구/비활성 5종 단위 테스트 통과.

### 3.3 주식 "skip→큐" 는 왜 단계 롤아웃(보류)인가

세 가지 이유로, 라이브 무장애 운영을 위해 **이번엔 강제 변경하지 않았다.**
1. **세션 게이트 한계:** 늦게 실행되는 KR 스캔이 15:30을 넘기면 어차피 `SKIPPED_SESSION`으로 정당하게 폐기된다. 큐잉의 이득은 **세션 내 지연**(예: 12:30 슬롯이 13:25에 실행)에 한정 — 이건 가치가 있지만, **근본 원인(느림) 해소(§3.2)가 더 결정적**이다.
2. **신규 데몬 위험:** 코인엔 큐 워커가 있으나 **주식 전용 워커는 없다.** factory dispatch를 구동하는 새 데몬 + cron 전면 전환 + systemd 유닛을 **테스트 없이** 라이브에 투입하는 것은 "방어적" 원칙 위반.
3. **이미 직렬 큐 인프라 존재:** `task_orchestrator`는 KR/US 엔진을 이미 지원. 즉 향후 활성화는 "새로 만들기"가 아니라 "워커 1개 추가 + cron 전환"으로 **단계적으로** 안전히 가능(§5 로드맵).

## 5. 신규 리스크 꼼꼼 검토 (가장 중요)

### R1. 서킷 브레이커 오트립(false trip) — 신선 데이터 누락 위험
- **시나리오:** 임계(40)가 너무 낮으면, 일시적 네트워크 깜빡임에도 트립 → 그 슬롯이 전일 데이터로 판정.
- **완화:** 기본 임계 40 + 윈도 60s는 "지속적 폭풍"에서만 걸리도록 보수적. 빈 응답/상장폐지는 카운트 제외. `..._FAILS` 상향으로 더 둔감하게.
- **모니터:** 로그 `UPSTREAM BREAKER TRIPPED` 빈도. 자주 뜨면 임계 상향 또는 업스트림 점검.
- **킬스위치:** `MARKET_DATA_UPSTREAM_BREAKER=0`.

### R2. 코인 무한 defer로 인한 적체(backlog) — 워커 미가동 시 좀비
- **시나리오:** `BITGET_ENQUEUE_ON_YIELD=1`인데 큐 워커가 안 떠 있으면, 인라인 cron이 적재만 하고 아무도 안 꺼냄 → PENDING 적체.
- **완화:** dedupe로 동일 모드 중복 적재 차단(무한 증식 없음). **활성화는 워커 가동 확인 후에만**(§2.2). `backlog_stats()` + `worker_heartbeat_age_sec()` 워치독으로 감시.
- **세션 폐기 부재 주의:** 코인은 24×7라 defer가 길어져도 의미가 있지만, 너무 오래된 진입 신호가 뒤늦게 체결되면 **stale entry** 가능. 필요 시 `BITGET_YIELD_DEFER_BACKOFF_SEC` 조정 + 워커에서 과도 노후 작업 폐기 정책 추가 검토.

### R3. defer가 attempts를 소모하지 않음 → 진짜 고장과 구분 필요
- defer는 의도적으로 attempts 중립. 따라서 **yield(정상 대기)** 와 **실제 실패(FAIL)** 는 명확히 분리돼야 한다. 현재는 `skipped_session + "yield_to_factory"` 문자열로만 defer하고, 그 외 실패는 기존 `fail()`(attempts 소모)로 간다 → 구분 OK. 단 향후 skip 사유 문자열을 바꾸면 이 매칭(`"yield_to_factory"`)도 함께 갱신해야 함(회귀 포인트).

### R4. busy_timeout 60s 대기 = 호출 지연 전가
- 락 경합이 심하면 writer가 최대 60초 블록될 수 있다(죽지 않고 기다림). 단일 트랜잭션이 60초를 넘기면 여전히 `database is locked` 발생 가능 → 그건 **트랜잭션이 너무 길다는 신호**(배치 분할/체크포인트 점검 대상). 60s는 SSOT 정책값이라 낮추지 말 것.

### R5. WAL `SQLITE_BUSY_SNAPSHOT` 은 busy_timeout으로 안 풀림
- 읽기 트랜잭션 도중 쓰기 승격 시 발생하는 즉시 오류는 busy_timeout이 재시도하지 않는다. 현재 핫 경로는 read/write 연결을 분리(RO는 `mode=ro`)하므로 노출 적음. 잔존 시 해당 지점만 짧은 재시도 래핑 검토.

### R6. 서킷 브레이커는 프로세스 전역(스레드 공유) — 다중 프로세스엔 독립
- 같은 스캔 프로세스의 30 스레드는 상태를 공유(정상). 단 KR/US/배치가 **별도 프로세스**면 브레이커도 독립(각자 카운트). 의도된 동작(프로세스별 보호)이며, 전역 공유가 필요하면 파일/DB 기반으로 승격 가능(현재는 불필요).

### R7. 동작 변경의 "기본 ON" 두 가지 재확인
- busy_timeout 통일(§1): "더 오래 기다림"일 뿐이라 **정합성 위험 없음.**
- 브레이커(§3.2): **장애일 때만** 작동. 정상 거래일엔 단 한 번도 개입하지 않음. → 둘 다 기본 ON 정당.
- 진짜 동작을 바꾸는 코인 큐(#2)만 **기본 OFF**로 묶어둠.

## 6. 롤백
- 브레이커: `MARKET_DATA_UPSTREAM_BREAKER=0`.
- 코인 큐: `BITGET_ENQUEUE_ON_YIELD=0`(기본값).
- busy_timeout: 코드 추가형이라 환경변수 토글 없음 — 문제 시 해당 `apply_busy_timeout` 라인 revert(단, 값이 60s로 늘어난 것뿐이라 부작용 보고 가능성 매우 낮음).

## 7. 권장 단계 롤아웃 순서
1. **(완료/ON)** #3 busy_timeout + #1 브레이커 — 모니터링하며 1~2일 관찰.
2. 코인 큐 워커 데몬 상시 가동 + `backlog_stats` 워치독 → `BITGET_ENQUEUE_ON_YIELD=1` (코인부터, 위험 낮음).
3. 안정 확인 후, 주식 통합 직렬 워커(KR/US dispatch) 추가 → cron `--enqueue` 전환 → 코인·주식 **단일 직렬 라인**으로 통합(동시 write 원천 소멸). 이 단계는 별도 테스트 후 진행.

## 8. 결론
- 청산 실패의 직접 원인(`database is locked`)은 **busy_timeout 통일**로 즉시·안전하게 완화(ON).
- 주식 스캔 60분 락 점유는 **장애 한정 로컬 빠른 폴백**으로 근본(느림) 완화(ON, 정상 불변).
- 코인 스캔 증발은 **defer 큐**로 해결하되, 라이브 안전을 위해 **워커 가동 후 플래그로 활성화**.
- 주식 큐잉은 세션 게이트 한계와 신규 데몬 위험 때문에 **근본 속도 해소를 우선**하고 큐는 단계 롤아웃으로 남김.
- 모든 위험 변경은 플래그 OFF가 기본이라 **이번 배포로 기존 로직이 꼬일 가능성은 최소화**되어 있다.

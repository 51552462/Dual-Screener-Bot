# [1/9] 거시 국면 및 국고(Treasury) 현황 — 업데이트·국고·가상청산누적 로직 전수 검토

> 사용자 질문 3가지에 대한 결론 요약
> 1. **"진짜 업데이트가 되고 있나?"** → 국면(regime)·켈리(kelly)는 **동적 갱신 O**.
>    국고(설정)는 **갱신 X(정적 상수)**. 가상 청산 누적은 **리포트마다 재계산 O(단, 윈도우 의존)**.
> 2. **"국고 3억이 계속 그대로, 돈 쓰는데 업데이트 안 됨"** → **정상(설계상 고정)**.
>    `CENTRAL_TREASURY_KR/US` 는 시스템 부팅 시 3억으로 1회 주입될 뿐, **거래 손익으로 이 값을
>    갱신하는 코드가 코드베이스 어디에도 없다.** 즉 라이브 NAV가 아니라 '고정 기준 자본(SSOT)'이다.
> 3. **"가상 청산 누적이 합산인가, 한 건인가?"** → **합산(SUM)**.
>    해당 시장 축의 청산행 전체에 대해 `Σ(노출액 × 청산수익률%)` 를 더한 값. 한 건이 아니다.
>    단, '노출액'이 대부분 **40만 원 고정 폴백**이라 3억 국고와는 수학적으로 분리돼 움직인다.
>
> → 사용자가 느낀 "디테일한 퀀트 로직이 안 잡혀 있다"는 정확하다. **국고(자본)와 실현손익을
> 잇는 자본 곡선(NAV/Equity)이 [1/9] 블록에는 존재하지 않는다.** 둘은 의도적으로 '분리(split)'돼
> 있고, 누적은 평면 40만 노출의 단순 합이라 복리/자본증감 의미가 없다.

## 4. "가상 청산 누적(켈리 노출)" = 합산(SUM)이다 — 한 건이 아니다

```206:221:reports/report_state_binder.py
def _treasury_from_ledger(df_closed_real, *, zero_invest_fallback, market="KR") -> float:
    """청산 행만 — weekly_flow_pnl SSOT(coalesce) 합."""
    from weekly_flow_pnl import dataframe_realized_pnl_sum
    if df_closed_real is None or df_closed_real.empty:
        return 0.0
    return dataframe_realized_pnl_sum(df_closed_real, market=market, zero_fallback=zero_invest_fallback)
```

`dataframe_realized_pnl_sum` 은 **윈도우 내 청산행 전부의 행별 손익을 더한다**:

```157:168:weekly_flow_pnl.py

    ...
    total = float(add_realized_pnl_column(df, market=market, zero_fallback=fb)["_realized_pnl"].sum())
```

행별 손익 산식(`row_realized_pnl`):

```71:87:weekly_flow_pnl.py
def row_realized_pnl(row, *, market="KR", zero_fallback=None) -> float:
    """청산 1건 실현 손익 = notional × final_ret%."""
    ...
    notional = row_notional(row, market=market, zero_fallback=zero_fallback)
    return notional * ret_v / 100.0
```

노출액(`notional`) 우선순위:

```45:68:weekly_flow_pnl.py
def row_notional(row, *, market="KR", zero_fallback=None) -> float:
    """투입 노출액: sim_kelly_invest → invest_amount → 시장 폴백."""
    # sim_kelly_invest > 0 → 사용
    # else invest_amount > 0 → 사용
    # else 폴백(40만 원)
```

### 그래서 화면의 큰 +숫자의 정체

```
가상 청산 누적 = Σ_{청산행 in 윈도우} ( 노출액 × final_ret% / 100 )
```

- **윈도우**: 일일 리포트는 **90일 롤링**, 주간은 7일.
- **노출액**: `sim_kelly_invest`/`invest_amount` 가 0/누락이면 **40만 원 평면 폴백**.
  실전에서 대부분 폴백이 잡히므로, 사실상 `Σ (40만 × 수익률%)` 에 가깝다.
- 예: 90일간 청산 60건, 평균 +3%면 ≈ `60 × 400,000 × 0.03 = 720만 원` 식으로 누적.
  **여러 건의 합이다. 한 건에서 나온 게 아니다.**

## 5. 왜 국고(3억)와 누적(+큰돈)이 따로 노나 — 의도된 '분리(split)'

```224:248:reports/report_state_binder.py
def _merge_treasury_policy(treasury_config_raw, ledger_realized_est, oms_equity, sys_config) -> str:
    """이중 계상 방지: 설정 국고와 장부 누적을 명시적으로 분리; 합산은 기본 안 함."""
    mode = ... or "split"          # 기본 split
    if mode in ("sum", "sum_estimate", "nav_estimate"):
        est = treasury_config_raw + ledger_realized_est   # 옵션일 때만 참고 NAV
    core = "※ 설정 국고는 구성 파일 SSOT. 가상 청산 누적은 청산행만 집계. ... 단순 합산하지 마세요."
```

기본 모드가 `split` 이라, 리포트는 **두 값을 더하지 않고 따로 표시**한다. 이유는 '이중 계상
방지'(만약 국고가 손익 반영분이면 더하면 중복). 하지만 현재는 국고가 **고정 상수**이고 누적은
**평면 40만 노출 합**이라, 둘 사이에 수학적 연결고리가 0이다. → 사용자가 느낀 "로직이 안 잡힘".

`REPORT_TREASURY_NAV_MODE=sum` 으로 두면 `참고 NAV=국고+누적` 한 줄이 추가로 뜨지만, 이는
어디까지나 '참고 추정'이고 진짜 자본 곡선은 아니다.

## 6. 구조적 결함 진단 (퀀트 관점)

1. **자본 곡선(NAV/Equity) 부재** — 국고(자본) → 포지션 사이징 → 청산 손익 → 국고 갱신으로
   이어지는 폐루프가 없다. 국고는 동결, 손익은 별도 집계. '돈을 쓰는데 잔고가 안 변한다'는
   체감의 근본 원인.
2. **평면 40만 노출의 의미 희박** — 누적이 실제 3억 자본의 켈리 사이징(`effective_kelly_risk`)을
   반영하지 않는다. 화면엔 "켈리 노출"이라 써 있지만 실제론 켈리와 무관한 40만 고정이 대부분.
3. **윈도우 의미 혼선** — 같은 "가상 청산 누적"이 일일=90일, 주간=7일로 윈도우가 다른데 라벨이
   동일해 '계속 쌓이는 총자산'으로 오해하기 쉽다(실은 롤링 합이라 오래된 청산은 빠져나감).
4. **주간엔 복리 곡선 도입했으나 일일 [1/9]엔 미적용** — 최근 `weekly_flow_pnl` 에 가상 자본
   곡선(`compute_virtual_equity_curve`, $E_t=E_{t-1}(1+f^*R_t)$)을 넣었지만, [1/9] 국고 블록은
   여전히 옛 `dataframe_realized_pnl_sum`(평면 합)을 쓴다. 두 리포트의 손익 정의가 불일치.

## 7. 개선 제안 (검토용 — 적용은 별도 지시 시)

- **P1. 라벨 정직화(무해·우선)**: "가상 청산 누적(켈리 노출)" → "가상 청산 누적(최근 N일 롤링,
  40만 평면 노출 가정)" 처럼 윈도우·노출 가정을 명시. 오해 해소.
- **P2. 켈리 연동 NAV 추정**: 누적을 40만 평면이 아니라 `국고 × effective_kelly_risk` 노출로
  재산식 → "이 자본·이 켈리로 굴렸다면" 의미가 생김. (주간의 가상 자본 곡선과 통일)
- **P3. 라이브 NAV 동기화기 신설**: 청산 손익을 `CENTRAL_TREASURY_*` 에 누적 반영하는
  동기화 잡(주간 결산 시 1회)을 만들면 국고가 실제로 증감. 단 이중계상 방지 위해 누적 표시를
  'NAV 내장'으로 전환 필요(설계 변경, 신중히).
- **P4. 일일/주간 손익 정의 통일**: [1/9]도 `compute_virtual_equity_curve` 기반 복리 Net PnL로
  교체하면 두 리포트가 같은 자본 곡선을 공유.

> P1은 즉시 적용 가능한 무해 패치다. P2~P4는 자본·손익 정의를 바꾸는 정책 변경이라, 어떤
> 의미(고정 기준자본 유지 vs 라이브 NAV 전환)를 원하는지 결정 후 진행하는 게 안전하다.

## 9. 한 줄 답변

> **국고 3억이 안 변하는 건 버그가 아니라 설계(고정 기준 자본·갱신 코드 없음)다.**
> **가상 청산 누적은 한 건이 아니라 최근 90일(일일)/7일(주간) 청산 전체의
> `Σ(노출액 × 수익률%)` 합산이며, 노출액은 대부분 40만 평면 폴백이라 3억 국고·켈리와
> 분리(split)돼 움직인다.** 국면·켈리는 MetaGovernor로 동적 갱신되지만, '자본 → 사이징 →
> 손익 → 자본'으로 잇는 NAV 폐루프가 [1/9]엔 없어서 "디테일 로직이 안 잡힌" 것처럼 보인다.

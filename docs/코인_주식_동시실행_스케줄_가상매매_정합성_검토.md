# 코인(Bitget) ↔ 주식(KR/US) 동시 실행 스케줄 & 가상매매 정합성 전체 검토

> 작성: 2026-06-29 (KST) · 범위: 코인을 한국/미국 퀀트가 "100% 돌아가는" 크론·시간대에
> 같이 돌려도 되는지, 아니면 진짜 유휴 시점에만 돌릴지에 대한 의사결정 검토.
> 상태: **read-only 검토 (코드 미수정)**
> 선행 문서: `bitget/docs/cron_timezone_db_lock_review.md` (2026-06-28, DB 락 원인 분석)

---

## 0. 이 문서가 답하려는 질문

사용자 질문 그대로:

1. 한국·미국 구조가 100% 돌아가는 크론/시간대에 **코인이 같이 돌아가도 되는가?**
2. 아니면 **진짜 아무것도 안 돌아가는 유휴 시점에만** 코인을 돌리는 게 나은가?
3. 코인이 "돌아가다 안 돌아가다" 하면 **가상매매의 데이터 축적·오픈/청산이 꼬일 수 있는가?**
   주식(KR/US)도 마찬가지 우려.

결론부터: **시간대 겹침 자체는 오픈/청산을 꼬이게 만들지 않는다.** 꼬이게 만드는 건
(a) 스캔을 **스킵(drop)** 하는 현재 방식과 (b) **DB 동시 writer 경합**이다.
그리고 "유휴 시점에만 코인 돌리기"는 **청산(track)이 24/7이어야 하므로 구조적으로 불가능**하다.
아래에서 근거를 단계별로 제시한다.

## 1. 오늘 확증된 증상 (스크린샷)

`scan_kr_ema5` 가 `SKIPPED_LOCK` 으로 증발:

```
[Factory Job] scan_kr_ema5 · SKIPPED_LOCK
run_id : 20260629T123055-06ae6903
window : 2026-06-29 12:30:55 → 13:25:56   (약 55분 대기 후 포기)
holder : holder_mode=scan_kr_dante pid=2505989 alive=True
         since=2026-06-29 12:25:29 age=3626s
file   : /home/ubuntu/dante_bots/Dual-Screener-Bot/.factory_runtime.lock
```

해석:
- `dante` 가 글로벌 factory 락(`.factory_runtime.lock`)을 **3626초(약 60분)** 점유.
- `dante` 슬롯은 원래 11:40인데 락을 잡은 시각이 12:25:29 → **이미 45분 지각 시작**(앞 슬롯 적체).
- `ema5` 슬롯은 12:30에 떠서 55분(`SCAN_LOCK_WAIT_SEC`) 대기하다 13:25:56에 **포기 → 증발**.

→ 이전 채팅에서 분석한 **"스캔 자체가 12배 느려짐(업스트림 FDR/yfinance 지연 + 동시성 2 세마포어 병목)"** 이
글로벌 단일 락을 타고 **연쇄 지각 + 슬롯 스킵**으로 번진 것이 그대로 관측됨.

관련 SSOT:
- `factory_scan_schedule.py` L19 `SLOT_INTERVAL_MINUTES = 50`, L23 `SCAN_LOCK_WAIT_SEC = 50*60+300 = 3300s`
- `factory_runtime.py` L356 `factory_job_lock(...)` — 단일 글로벌 flock
- `system_auto_pilot.py` L2368~2371 — scan 모드 lock_timeout = `SCAN_LOCK_WAIT_SEC`


> 중요: KR·US 스캔은 **같은 글로벌 락**(`.factory_runtime.lock`)을 공유하므로 서로 절대 동시 실행 안 됨.

### 2-4. 진짜 "유휴(idle)" 창은 얼마나 되나?

주식(KR 08:00~18:45 + US 22:00~07:00)을 빼면 **평일 유휴는 사실상**:
- **약 19:00 ~ 22:00 (KST)** — 단, 18:45 daily-kr 가 한동안 돌므로 실효 19:30~22:00 (~2.5h)
- **약 07:00 ~ 09:45 (KST)** — 단, 06:45 daily-us, 08:00 KR data-refresh 가 끼어 실효 ~1.5h

→ **하루 유휴는 다 합쳐도 ~4시간**. (주말은 비교적 여유, 단 토 10:00/10:05 잡 존재)

**이것이 1번 질문의 핵심 제약이다: 코인 전체(스캔 19슬롯 + 24/7 추적)를 4시간 유휴에 욱여넣는 것은 불가능하다.**

## 3. 가상매매 생명주기 — "오픈"과 "청산"은 서로 다른 트리거다

이게 "꼬임" 판단의 핵심이다. **오픈은 스캔이, 청산은 추적(track)이 담당하며 서로 분리돼 있다.**

### 3-1. 코인(Bitget)

| 행위 | 담당 | 트리거 | yield 대상? |
|---|---|---|---|
| **오픈(신규 진입)** | `scan_*` → `run_master_scan` | 스캔 슬롯(분산) | **예 (스킵 가능)** |
| **청산/추적** | `track_positions` → `track_daily_positions` | `*/15` 상시 + 각 스캔 tail | **아니오 (항상 실행)** |
| 정산 점검 | `reconcile` | `:53` 매시 | 아니오 |
| 데이터 갱신 | `data_refresh` | `43 */4` | **예 (스킵 가능)** |

근거:
- `bitget_schedule_guard.py` L29 `_YIELD_GATED_MODES = ("scan_", "data_refresh")`
  → **track / reconcile / watchdog / health 는 양보 안 함 = 주식과 무관하게 항상 실행.**
- `bitget/pipelines/bitget_pipelines.py` L457 `_pipeline_track_positions` = track_spot + track_futures (critical), `*/15`.
- 청산 로직 `bitget/forward/ledger.py` L567 `track_daily_positions`:
  - OPEN 포지션을 읽어 `_get_latest_bar(...)` **최신 OHLCV 바**로 손절/타임스탑/타깃 판정.
  - 데이터 부족(`len(hist_df) < 20`)이면 **닫지 않고 건너뜀**(좀비 14일↑만 강제 청산) → L598~604.

**핵심 결론(코인):** 스캔을 스킵해도 **이미 열린 포지션의 청산은 막히지 않는다.**
청산은 `*/15` track 이 독립적으로 돌리기 때문. 스캔 스킵의 영향은 **"그 슬롯의 신규 진입을 놓침"** 뿐.

### 3-2. 주식(KR/US)

- **오픈**: 각 스캐너가 시그널 포착 시 `forward_trades`(가상장부)에 등재(ENROLLED) — `scanner_funnel.py` 의 pipeline_line 참조.
- **청산/추적**: `daily_audit`(KR 18:45 / US 06:45)의 track 스텝(`track_daily_positions`)이 일 1회 마크/청산.
  - 즉 주식은 코인처럼 `*/15` 가 아니라 **하루 1회(장 마감 후)** 추적 사이클.

**핵심 결론(주식):** 스캔(슬롯)이 SKIPPED_LOCK 으로 증발하면 **그 스캐너의 그날 신규 진입이 누락**된다.
청산은 daily_audit 가 돌면 정상. 단 daily_audit 자체가 같은 글로벌 락/지연에 막히면 청산도 밀린다.

## 4. "돌아가다 안 돌아가다"가 실제로 꼬이게 만드는 지점

사용자 직관("on/off 하면 꼬인다")은 **부분적으로 맞다.** 정확히 어디가 꼬이는지 분리하면:

### 4-1. 꼬이지 않는 것 (오해 해소)
- **시간대 겹침 그 자체**: 오픈/청산이 서로 다른 트리거라, 코인 스캔이 주식과 같은 시각에 떠도
  오픈/청산 "정합성"이 깨지진 않는다.
- **스캔 스킵 → 청산 막힘**: 안 막힌다. 청산은 코인 `*/15`, 주식 daily_audit 가 독립 수행.

### 4-2. 진짜로 꼬이는 것 (실제 리스크)
1. **엔트리 표본 편향 (코인·주식 공통).**
   스캔이 어떤 날은 돌고 어떤 날은 스킵되면, 특정 스캐너/특정 장세의 **진입 표본이 들쭉날쭉** 쌓인다.
   → forward/shadow 성과·진화튜닝 데이터가 **불균등 샘플링**으로 편향. (데이터 "축적이 꼬인다"의 실체)

2. **DB 동시 writer 경합 → 청산(track) 실패 (코인).**
   선행 문서(`cron_timezone_db_lock_review.md`)가 규명: `data_refresh` 별도 락 + 24/7 WS/스나이퍼/스냅샷
   데몬 + yield 비대상 `track_positions` 가 같은 `bitget_market_data.sqlite` 에 동시 write →
   `database is locked` → **track 이 실패하면 그 회차 청산/마크가 누락**. **이게 오픈/청산이 실제로 꼬이는 주경로.**

3. **데이터 staleness → 청산 지연 (코인).**
   `data_refresh` 가 주식에 양보(스킵)되어 OHLCV 가 신선하지 않고 WS 도 끊기면,
   `track_daily_positions` 가 "데이터 부족"으로 청산을 **연기**(L601~604) → 닫혀야 할 포지션이 **잔류** →
   보유기간/손익 통계 왜곡.

4. **글로벌 락 연쇄 지각·스킵 (주식).**
   §1 처럼 한 스캔이 느리면 뒤 슬롯이 줄줄이 지각/SKIPPED_LOCK → 그날 진입 누락 + 스케줄 붕괴.
   여기에 더해, 코인 가드는 factory 락이 **90분(`BITGET_FACTORY_YIELD_MAX_AGE_SEC=5400`) 넘으면 좀비로 간주해 양보 중단**
   (`bitget_schedule_guard.py` L92~94) → 주식이 적체로 락을 오래 쥐면 **코인이 양보를 멈추고 동시 가동** →
   4GB 서버 자원·DB 경합 가중 → 악순환.

## 5. 질문에 대한 답

### Q1. 코인을 주식 100% 활성 시간대에 같이 돌려도 되나?
**원칙적으로 된다. 단 현재 "스킵(yield-to-factory)" 방식과 DB 경합을 고치는 조건에서.**
- 겹침 자체는 오픈/청산 정합성을 깨지 않는다(§4-1).
- 다만 지금처럼 **겹칠 때 스캔을 drop** 하면 엔트리 표본이 편향되고(§4-2-1), DB 경합으로 track 이 실패한다(§4-2-2).

### Q2. 그냥 진짜 유휴 시점에만 코인을 돌리는 게 낫나?
**전체 코인 시스템을 유휴에만 돌리는 건 불가능하다.**
- 청산(track `*/15`)과 reconcile, WS ingest 는 **24/7이어야 포지션이 닫히고 마크된다.** 유휴(~4h)로 제한하면
  청산이 멈춰 포지션이 잔류 → 오히려 더 크게 꼬인다.
- "스캔(=신규 진입)만 유휴창에 몰기"는 가능은 하나, 24/7 암호화폐의 진입 커버리지를 ~4h로 깎고
  여전히 24/7 WS/스냅샷 데몬과의 DB 경합(청산 실패의 진짜 원인)은 그대로 → **이득 대비 손실이 큼.**

### Q3. on/off 가 가상매매를 꼬이게 하나?
**"스캔 스킵" 자체보다, 스킵을 유발하는 두 구조가 꼬이게 한다:**
- 코인: DB 동시 writer 경합(track 실패·청산 지연) + data_refresh 양보로 인한 staleness.
- 주식: 느린 스캔 → 글로벌 단일 락 연쇄 지각/스킵 → 진입 누락.

## 6. 권장 방향 (코드 수정 없이 — 제안만)

우선순위 순. 실제 적용은 **승인 후 별건**으로.

1. **[코인] 스캔 충돌을 "스킵" 대신 "큐 대기"로 전환.**
   이미 존재하는 단일 직렬 워커(`bitget/pipelines/queue_worker.py`, `task_orchestrator.py`)로
   cron → `--enqueue` 시키면, 주식과 겹쳐도 **drop 대신 PENDING 대기 후 순차 실행** →
   **차수 누락 0 + 동시 write 소멸**. (근거: `bitget/docs/infra_next_steps_a_b_plan.md` (b)안)

2. **[코인] DB writer 직렬화 + busy_timeout 통일.**
   `data_refresh` 별도 락 / 24/7 WS·스나이퍼·스냅샷 데몬 / yield 비대상 track 이 같은 DB 동시 write →
   `database is locked`. 전 모듈 `PRAGMA busy_timeout` 통일(30~60s) + WAL 진입 락 회피 +
   공용 write 락으로 직렬화. (근거: `cron_timezone_db_lock_review.md` PART C / 다음단계 (a)(b)(c))

3. **[주식] 글로벌 락 연쇄 지각의 뿌리 = 느린 스캔 해소.**
   (이전 채팅 분석) KR 데이터 fetch 체인이 **네트워크 우선(FDR→yfinance→로컬 SQLite)** + 전역 동시성
   2(`market_data_fetcher.py` L42 `_MAX_CONCURRENT_UPSTREAM=2`)인데 스캐너는 30스레드 →
   업스트림 저하 시 12배 지연 → 60분 락 점유 → ema5 등 스킵.
   → 로컬 SQLite 우선 읽기/업스트림-스레드 정합/락-스킵→대기 전환 검토.

4. **[양쪽] 청산을 시간대 겹침과 무관하게 보호.**
   track(코인 `*/15`, 주식 daily_audit)은 **항상 자원/DB를 확보**해야 한다. 1·2번이 적용되면 자연 해소.

5. **[운영 관찰] 좀비 양보 컷오프 점검.**
   `BITGET_FACTORY_YIELD_MAX_AGE_SEC=5400`(90분)은 주식 락이 오래 걸리면 코인 양보를 끊는다.
   주식 락 적체가 잦으면 이 값이 동시 가동을 유발하므로, 3번(주식 락 적체) 해결 전엔 모니터링 필요.

## 7. 한 줄 요약

- **"코인을 주식 시간대에 돌려도 되냐"의 답은 시간표 문제가 아니다.** 오픈(스캔)과 청산(track)은
  분리돼 있어 **겹침 자체는 정합성을 깨지 않는다.**
- **진짜 꼬임은 ① 스캔을 drop 하는 방식(엔트리 편향) ② DB 동시 writer 경합(청산 실패·지연)** 이다.
- **유휴 시점에만 코인 돌리기는 불가능**하다(청산은 24/7 필수, 유휴는 ~4h뿐).
- 올바른 방향: **겹쳐도 안전하게 — "스킵 → 큐 대기" + DB writer 직렬화/busy_timeout 통일**, 그리고
  주식 쪽은 **느린 스캔(60분 락 점유)** 의 뿌리를 풀어 연쇄 스킵을 없애는 것.

> 본 문서는 검토만 했고 **코드는 일절 수정하지 않았다.** 확정 검증은 서버 로그
> (`logs/factory_scan_kr_*.log`, bitget `database is locked` 알람, `ops_logger` 게이지)로 가능.

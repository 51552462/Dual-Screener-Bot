# 한국·미국·코인 100% 가동 — 통합 점검 & 수정 필요사항 (코드 미수정)

작성일: 2026-06-29
범위: 주식 factory(KR/US) + 코인(Bitget) 동시 100% 가동 시 문제·리스크·활성화 누락 전수 점검.
원칙: **여기서는 수정하지 않는다.** 발견·근거·권장 조치만 기록. 우선순위(P0=가동 막힘/데이터 오염, P1=효과 미발현/숨은 위험, P2=정합/관측 개선).

## 0. 한눈에 (요약 표)

| # | 영역 | 증상/리스크 | 우선순위 | 한 줄 조치 |
|---|---|---|---|---|
| A | 코인→주식 canary | 익스포터 **4시간 주기** vs 소비자 **90분 신선도 게이트** → 국면잡 시점에 거의 항상 stale 무시 | **P0** | 전용 `*/15` canary cron 추가 또는 스캔 prelude에 훅 추가(택1) |

## 1. [P0] 코인 선행 레이더(canary) — 주기 ↔ 신선도 게이트 충돌

### 근거
- 코인 크론(`bitget/deploy/generate_bitget_crontab.py`)의 데이터 리프레시: `43 */4 * * *` = **4시간마다(하루 6회)**.
- canary 익스포터는 `bitget/pipelines/bitget_pipelines.py`의 `_pipeline_data_refresh` **tail에만** 배선됨(스캔 prelude의 `data_refresh_incremental`에는 없음) → 결국 **4시간 주기로만** 갱신.
- 소비자(`predictive_regime_ensemble._load_crypto_canary`)는 `CANARY_MAX_STALE_SEC=5400`(**90분**) 초과 시 무시.
- 소비자 실행 시점: `run_and_evolve`는 `regime_meta_analyzer.analyze_market_regime` → `meta_learner.run_meta_learning_cycle` 경로로 **일일 audit/메타 사이클**에서 호출(예: `--daily-kr` 18:45, `--daily-us` 06:45 KST).

### 문제
4시간 주기 갱신인데 게이트는 90분. 일일 국면잡(18:45/06:45)에서 가장 가까운 canary는 보통 **120분+ 전** → **거의 항상 stale로 폐기** → canary가 국면에 사실상 안 실린다. **선행 레이더가 무력화됨.**

### 권장 조치 (택1·수정 보류)
1. **전용 cron 추가(권장)**: `*/15 * * * * … python -m bitget.canary_exporter`(또는 `bitget.sh --canary` 모드 신설) → 10~15분 신선 유지. 경량(공개 API 5~10콜).
2. **스캔 prelude 훅 추가**: `_prelude_full/_prelude_light`의 `data_refresh_incremental` 뒤에 `canary_export` 추가 → 24×7 스캔 슬롯마다 갱신.
3. **빠른 완화**: `CANARY_MAX_STALE_SEC`를 4h 주기보다 크게(예: 5h=18000) — 단 stale 신호를 그만큼 허용하므로 1·2가 정석.

## 2. [P0] canary 파일 경로 — 크로스 프로세스 해석 불일치 위험

### 근거
- 익스포터(코인 프로세스)는 `bitget.infra.data_paths.canary_state_path()` = `bitget_data_dir()/bitget_canary_state.json`에 기록.
- `bitget_data_dir()` 우선순위: ① env `BITGET_DB_STORAGE_PATH` → ② `bitget_system_config.json`의 키 → ③ 레거시 `bitget/` 패키지 디렉터리.
- 소비자(주식 factory 프로세스)도 동일 함수를 import해 경로를 **재계산**한다.

### 문제
주식 factory 프로세스의 환경에 `BITGET_DB_STORAGE_PATH`가 **없고**, 코인은 `bitget/.env`로만 그 값을 받는 경우 → 두 프로세스가 **서로 다른 디렉터리**를 가리켜 주식이 canary를 못 찾는다(조용히 `(0.0, False)`로 graceful → 무영향이지만 **기능이 안 켜짐**).
- `deploy_bitget_factory.sh`도 "ensure `bitget/.env` … `BITGET_DB_STORAGE_PATH`"만 안내 → 주식 .env에는 없을 수 있음.

### 권장 조치
- **두 프로세스 모두**에 `BITGET_CANARY_STATE_PATH=/abs/경로/bitget_canary_state.json`를 **명시**(가장 확실). 소비자 `_load_crypto_canary`와 익스포터 `canary_state_path()` 모두 이 env를 1순위로 읽도록 이미 설계되어 있음.
- 또는 `BITGET_DB_STORAGE_PATH`를 루트 `.env`(주식·코인 공통 로드)로 통일.

## 3. [P1] OI 수집 실측 미검증 — 유동성 스트레스가 0 고착 가능

### 근거
- 로컬 검증에서 동적 알트 선정·펀딩·BTC는 실값을 받았으나 **`oi_total_usdt=null`**(거래소 `fetch_open_interest` 응답 형식/마켓 타입 이슈로 추정).
- OI 24h 변화율은 **자체 링버퍼**(`bitget_canary_oi_history.json`) 차분이라 **최초 24h 누적 전에는 None**(콜드스타트) → `oi_drop=0` → `stress=0`.

### 문제
OI가 null/0이면 `crypto_liquidity_stress`(기하평균)가 **항상 0** → 게이트(0.8) 영구 미충족 → 지표 ① 사실상 비활성.

### 권장 조치
- 서버에서 `python -m bitget.canary_exporter` 1~2회 실행 후 JSON의 `oi_total_usdt`/`oi_total_24h_change_pct`가 채워지는지 확인.
- 필요 시 OI 호출을 `fetch_open_interest_history(sym,'1h',limit=25)` 기반으로 보강(즉시 24h 변화 산출 → 콜드스타트 제거). (설계 §2.1의 1순위안)

## 4. [P1] 코인 "스킵→대기"(#2) 휴면 — 활성화 누락

### 근거
- `dante-bitget-queue-worker.service`는 `deploy_bitget_factory.sh`가 **enable**(상시 가동)함. ✅
- 그러나 크론 기본은 **inline 실행**(`generate_bitget_crontab.py` `use_queue=False`) → 워커에 들어오는 작업 없음.
- `BITGET_ENQUEUE_ON_YIELD` 기본 **0** → yield 시 defer 미작동.

### 문제
"주식 바쁠 때 코인 스캔 대기 후 순차 실행"이 **켜져 있지 않음**(설계대로 안전 기본값). 즉 현재는 여전히 yield=Drop.

### 권장 조치 (검증 후 단계 활성화)
1. `python bitget/deploy/generate_bitget_crontab.py --use-queue` → `install_bitget_cron.sh` 재설치.
2. 큐 워커 가동 확인: `systemctl status dante-bitget-queue-worker` + `task_orchestrator.backlog_stats()`.
3. `BITGET_ENQUEUE_ON_YIELD=1` 설정 후 코인 서비스 재시작.
4. 며칠 PENDING/oldest_pending_age 관측.

## 5. [P1] 업스트림 브레이커(#1) — 부분 저하 구간엔 여전히 느림

### 근거·문제
- 브레이커는 **윈도 60s 내 40건** 임계를 넘겨야 트립(기본 ON). 그 **미만의 만성적 저하**(예: 종목당 1~2회 timeout)에서는 트립 안 함 → `_MAX_CONCURRENT_UPSTREAM=2` 세마포어 + 3회 재시도 백오프가 남아 스캔이 여전히 늘어질 수 있음.
- KR 스캔 슬롯 간격은 보통 ~50~70분(10:00→10:50→11:40…). 한 스캔이 그 이상 끌면 다음 슬롯 SKIPPED_LOCK 재발 가능.

### 권장 조치
- 서버 관측 로그(`UPSTREAM BREAKER TRIPPED` 빈도, 슬롯별 소요시간)로 임계(`MARKET_DATA_UPSTREAM_BREAKER_FAILS`)·윈도·쿨다운 튜닝.
- 만성 저하가 잦으면 `_MAX_CONCURRENT_UPSTREAM` 상향(429 위험과 트레이드오프) 또는 KR 일중 OHLCV 사전 적재 강화 검토.

## 6. [P1] 주식 잔존 DB 락 표면 — 핫 writer 외 다수 미적용

### 근거
- 이번 #3에서 busy_timeout을 적용한 곳: `low_ram_sqlite_pragmas`(공용), `forward/shared.py`(가상매매 장부), `data_updater.py`(OHLCV), `market_data_fetcher.py`.
- 그러나 저장소 전반에 `sqlite3.connect`가 **수십 개**(상당수 기본 `timeout=5s`). 대부분 reader라 WAL에서 writer를 막진 않지만, **08:00 data_refresh 대량 write 중** 또는 메타/리포트 동시 write 시 5s 만에 `database is locked` 가능한 지점 잔존(예: 일부 메타/리포트/스토어 writer).

### 권장 조치
- 전수 일괄 변경은 위험 → **market_data.sqlite/forward DB에 write 하는 모듈만** 우선 식별해 `apply_busy_timeout` 추가(읽기 전용은 후순위).
- 중기적으로 주식측에도 코인 `shared_db_connector` 같은 **단일 커넥터 SSOT** 도입 검토.

## 7. [P2] canary 거시(VIX) 소스 가용성

### 근거·문제
- 익스포터의 `_macro_up()`는 ① 코인 폴더의 `bitget_doomsday_status.json`(둠스데이 브릿지 산출) → ② yfinance `^VIX` 폴백.
- 주식 둠스데이 크론은 `0 17 * * 1-5`(평일 17:00 KST) 1회 → 주말/장중엔 갱신 안 됨. 미러 파일이 없거나 오래되면 ①실패.
- yfinance 폴백은 서버 인증서/네트워크에 의존(로컬에선 cert 오류로 실패함).

### 권장 조치
- 서버에서 `bitget_doomsday_status.json` 생성/주기 확인(둠스데이 브릿지 step이 코인 파이프라인에 도는지) 또는 yfinance `^VIX` 접근성 확인.
- 둘 다 불가하면 `macro_contagion_risk`는 항상 False → 지표 ② 비활성(안전하나 기능 미발현).

## 8. [P2] 자원(4GB) · 동시성

### 근거
- 주식 `MAX_WORKERS=1`(factory_resource_limits), 코인 서비스 `MemoryMax=2G`, 큐워커 `MemoryMax=2G`.
- 코인 WS 데몬(`dante-bitget-ws`) 24×7 DB writer + 스냅샷 타이머 5분 + 트랙 */15 + 워치독 */5.

### 평가·조치
- 코인은 factory yield(`BITGET_YIELD_TO_FACTORY=1`) + busy_timeout으로 충돌 완화됨(이번 작업 반영). 다만 **WS·스냅샷·트랙이 동시 write** 하는 순간은 남음 → busy_timeout 60s가 흡수.
- 큐워커까지 상시 가동 시 메모리 합(2G+2G+…)이 4GB에 근접할 수 있음 → 큐워커 활성화(#4) 전 `free -m`/OOM 로그 점검 권장.

## 9. [P2] 주식측 "스킵→큐" 부재 (이전 리포트 재확인)
- 코인은 defer 큐가 준비됐지만 **주식 전용 큐 워커는 없음** → 세션 내 SKIPPED_LOCK은 여전히 Drop.
- 세션 게이트 때문에 장 마감 후 실행은 어차피 무의미 → **근본은 #1(속도)**. 주식 큐는 별도 단계 롤아웃 권장(`task_orchestrator`가 KR/US 엔진을 이미 지원하므로 "워커 1개 + cron 전환"으로 확장 가능).

## 10. 이미 안전하게 처리된 것 (회귀 없음 확인)
- #3 busy_timeout 통일: "더 오래 기다림"이라 정합성 위험 0(핫 writer 적용 완료).
- #1 브레이커: 장애 시에만 작동, 끄는 스위치 존재.
- #2 코인 defer 큐: 코드/단위테스트 통과, **기본 OFF**라 미활성 시 영향 0.
- canary 소비자: **SHADOW 기본**(`CRYPTO_CANARY_PENALTY_ENABLED=0`) → 국면 점수 미수정, 로그만. 실패/노후 시 graceful.

## 11. 100% 가동 체크리스트 (운영자용)

**즉시(가동 차단 해소)**
- [ ] **A**: canary 전용 `*/15` cron 추가 또는 스캔 prelude 훅 추가(택1).
- [ ] **B**: 주식·코인 양쪽 env에 `BITGET_CANARY_STATE_PATH`(또는 공통 `BITGET_DB_STORAGE_PATH`) 동일 절대경로.

**검증(효과 확인)**
- [ ] **C**: 서버 `python -m bitget.canary_exporter` → JSON `oi_total_usdt` 채워지는지.
- [ ] **G**: `bitget_doomsday_status.json` 신선도 또는 yfinance `^VIX` 접근.
- [ ] 주식 국면잡 로그에 `[CANARY SHADOW] … applied=False est_score_delta=…` 출력 확인(며칠 관측).

**단계 활성화(검증 후)**
- [ ] **D**: 큐 모드 cron(`--use-queue`) + `BITGET_ENQUEUE_ON_YIELD=1`.
- [ ] canary: 관측 정합 확인 후 `CRYPTO_CANARY_PENALTY_ENABLED=1`.

**상시 운영 점검**
- [ ] **E**: `UPSTREAM BREAKER TRIPPED` 빈도·슬롯 소요시간.
- [ ] **F**: data_refresh(08:00) 중 `database is locked` 로그.
- [ ] **H**: 큐워커 활성화 전후 `free -m`/OOM.

## 12. 결론
- **현 상태로도 주식·코인은 가동되며 회귀 위험은 낮다**(위험 변경은 전부 기본 OFF/장애시 한정).
- 단, **선행 레이더(canary)의 실효를 보려면 A·B가 필수**(지금은 주기/경로 때문에 사실상 비활성).
- "스킵 제로(코인)"와 "청산 실패 제로"의 **완전한 효과**는 D(큐 활성화) + C/G(데이터 소스 검증)까지 가야 100% 발현된다.
- 권장 순서: **A·B 수정 → C·G 검증 → (SHADOW 며칠) → D·canary 활성화 → E·F·H 상시 관측.**

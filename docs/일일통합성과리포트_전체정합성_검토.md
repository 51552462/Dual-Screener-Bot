# 일일 통합 성과 리포트 — 전체 정합성·정체(Stagnation) 종합 검토

> 질문: *"일일 통합 성과 리포트 안에 들어가는 모든 내용이 진짜 하나씩 다 잘 맞물려 있고, 정체 안 되고 업데이트 잘 되고 다 돌아가는지 빠뜨리지 말고 전체 검토."*
>
> 검토 방식: 오케스트레이터(`forward/deep_dive.py:40 send_comprehensive_daily_report`)부터 시계 SSOT(`reports/daily_report_context.py`, `reports/report_timekeeper.py`), 선행 갱신 7종, 9개 섹션 각각의 데이터 소스·갱신 주체·0건 폴백·정체 위험을 **코드 라인 단위로 실측**.
>
> **종합 결론:** 9개 섹션 전부 매 리포트마다 `forward_trades`의 **롤링 윈도우를 라이브로 재계산**하는 구조라, 리포트 로직 자체에는 값이 굳는(frozen) 캐시가 없다. **정체의 유일한 진짜 원인은 "상류 데이터(OHLCV/청산)가 안 들어와 윈도우 표본이 0이 되는 것"**이며, 이는 이미 ① `data_refresh` 크론(08:00 OHLCV 벌크), ② `track_daily_positions`(청산→워터마크 전진), ③ 헤더의 `lag/워터마크` 투명 표기로 방어·관측되고 있다. 아래에 각 지점과 잔여 리스크를 정리한다.

## 1. 리포트 생성 파이프라인 (실행 순서 = 맞물림의 뼈대)

`factory.sh --daily-kr|--daily-us` → `_pipeline_daily_audit_kr/us`(`factory_pipelines.py:985,1007`). 순서가 핵심이다:

```
[프렐류드] 시장 세션 게이트 + ensure_report_pipeline_data(매크로/OHLCV 하이드레이트)
   │
1. track_daily_positions_kr/us   ← OPEN→CLOSED 청산 확정 (DB 청산 워터마크 전진의 원천)
2. deep_dive_kr/us               ← 실무자 딥다이브
3. doomsday_bridge_sync          ← DEFCON/INVERSE SSOT 동기화
4. reporter_cleanup_zombie       ← 유령 OPEN 정리
5. forward_trade_identity        ← 거래 식별/정규화
6. practitioner report
7. comprehensive_daily_report    ← ★ 본 리포트([0]~[9/9]) 송출
8. ai_overseer
```

핵심 인과: **1번이 돌아야 청산이 쌓이고 → 워터마크가 전진하고 → 7번 리포트의 윈도우 표본이 갱신**된다. 1번이 안 돌면 모든 섹션이 "표본 0"으로 정체된다. 1번은 `critical=True`(`:989`)라 실패 시 파이프라인이 멈춰 알린다.

추가로 `send_comprehensive_daily_report` 진입 직후 **자체적으로도** 선행 갱신을 한 번 더 수행한다(`deep_dive.py:61-109`): hydrate → meta heal → sector spillover → sentiment → doomsday bridge. 이중 안전망.

## 2. 시계 SSOT & 정체 판별의 핵심 — Timekeeper

모든 섹션은 `DailyReportContext`가 만든 **단일 시간 창**을 공유한다(`daily_report_context.py:44`).

| 개념 | 의미 | 산출 |
|---|---|---|
| `session_anchor` | "방금 마감된 영업일" (KR=KST 영업일, US=ET 16:00 기준 마감일) | `report_timekeeper.py:95,103` |
| `rolling_cutoff` | anchor − 90일(기본) | `:72` |
| `db_watermark_exit` | DB에 실재하는 **최신 청산 exit_date** | `daily_report_context.py:73` `query_latest_closed_trade_date` |
| `lag` | 워터마크가 anchor보다 며칠 뒤처졌나(영업일) **0=최신** | `report_timekeeper.py:133` |

→ **글로벌 헤더**(`:120`)와 **시장별 헤더**(`:136`)에 `워터마크·lag·롤링·읽기소스(MAIN/SNAPSHOT)`가 매번 찍힌다. 즉 **정체가 발생하면 리포트 스스로 `lag N · 워터마크 YYYY-MM-DD`로 자백**하게 설계돼 있다. lag가 0이 아니면 상류 데이터가 밀린 것.

윈도우 슬라이스(`:156 load_market_slice`)는 매번 `SELECT * FROM forward_trades WHERE sig_type NOT LIKE '%INCUBATOR%'`로 **풀 스캔 후 라이브 슬라이스** → 캐시 정체 없음. (market 컬럼 오기입 방어 위해 code 기반 정규화까지 수행 — `:171` 주석.)

## 5. 정체(Stagnation) 종합 진단

### 진짜 정체가 나는 유일한 경로
```
OHLCV 미갱신(08:00 data_refresh 실패) 또는 스캔 미작동
   → 신규 진입/청산 안 생김
      → track_daily_positions 가 닫을 게 없음
         → db_watermark_exit 동결 → lag↑
            → 윈도우 [cutoff, anchor] 안의 청산 표본 0
               → [2/3/5/6/7/9] 가 "표본 0/검증 부족"으로 정체
```
즉 **리포트는 정직하다.** 값이 안 변하면 그건 리포트 버그가 아니라 상류 데이터가 멈춘 것이고, 그 사실이 `lag·워터마크`로 즉시 노출된다.

### 이미 적용된 방어막 (정상 작동 확인)
1. **08:00 `data_refresh` 크론** — OHLCV 벌크 갱신(과거 동결 정체의 직접 해결).
2. **`track_daily_positions` critical 스텝** — 실패 시 파이프라인 중단·가시화.
3. **세션 게이트** — 장 마감 후에만 daily_audit 실행(장중 헛스캔 방지, "장외시간 스킵"의 정상 동작).
4. **lag/워터마크/읽기소스 헤더** — 매 리포트 자가 진단 표기.
5. **MAIN/SNAPSHOT 읽기소스 라벨** — 스냅샷 폴백 시 즉시 식별(`report_timekeeper.py:91`).
6. **선행 갱신 이중화** — 프렐류드 + 리포트 진입부 두 번.

### 잔여 리스크 (점검 권장)
- **R1. 조용한 스킵:** 선행 갱신 #3·#4·#6은 try-skip이라, 외부 API 장애가 누적되면 스테일인데도 리포트가 멀쩡해 보일 수 있다. → 로그 모니터링 필요(§6).
- **R2. 매크로 lookback 강등:** 라이브 매크로 실패 시 과거행으로 대체(`report_pipeline_hydrate.py:265`). 경고는 로그에만 남으므로 [1/9] 국면이 어제값일 수 있음.
- **R3. 400k 폴백 잔재:** [2/9] 리더보드(`deep_dive.py:299`)와 [3/9] 분석기(`:348`)가 아직 `zero_invest_fallback=400000`을 쓴다. Live NAV 엔진으로 이관한 주간 리포트와 달리 **일일 [2]/[3]은 구(舊) 고정폴백**이 남아 있어 복리 정합성이 약간 다르다(정체는 아니나 일관성 개선 여지).
- **R4. lag 임계 경보 부재:** lag가 커져도 "표기"만 하고 별도 텔레그램 경보는 없다. 임계(예: lag≥2) 시 능동 경보를 붙이면 정체를 더 빨리 포착 가능.

## 7. 결론

- **맞물림:** 9개 섹션 + 꼬리(Δ/Executive)가 **단일 윈도우(Timekeeper)·단일 슬라이스**를 공유하므로 섹션 간 표본 잣대가 일치한다. 구조적 정합성은 견고하다.
- **정체:** 리포트 로직에는 정체를 만드는 캐시가 없다. 정체는 오직 상류(OHLCV/청산) 공급 중단에서만 오고, 그것은 `lag/워터마크`로 자가 노출되며 `data_refresh`·`track_daily_positions`로 방어된다.
- **업데이트:** 매 실행 라이브 재계산 + 선행 갱신 7종(이중화). 정상 작동 중.
- **개선 여지(낮은 우선순위):** R1 로그 모니터링, R3 일일 [2]/[3] 400k 폴백 → Live NAV 이관, R4 lag 임계 능동 경보.

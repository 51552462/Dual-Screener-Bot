# 장세별(상승·횡보·하락) 로직 수익률·방어률 순위 — 구조 검토

> **질문:** "상승·횡보·하락장 때 장세마다 **가장 수익률이 좋고 방어률이 좋았던 로직 순위**를 매기는 부분이 있는가?"
> **범위:** KR·US 메인 구조 (bitget / legacy_archive 제외) · 코드 변경 없음 · 검토 전용
> **방법:** 코드 정독 + 직접 grep 확인 (`system_auto_pilot.py`, `meta_governor.py`, `regime_meta_analyzer.py`, `evolution/deathmatch_*`, `mab_capital_allocator.py`, `forward/shared.py`)

---

## 0. 한 줄 답

**아니오. "장세(regime)별로 로직(sig_type)을 따로 순위 매기는" 부분은 없습니다.**

- 장세 판별(regime detection)은 **있다**.
- 거래마다 진입 시점 장세 라벨(`forward_trades.entry_regime`)도 **저장된다**.
- **수익 순위표**(데스매치·콜로세움·MAB)도 **있다**.
- **방어 순위/점수**(MDD·연패·변동성 페널티)도 **있다** — 데스매치는 수익+방어를 한 점수로 합쳐 순위까지 낸다.
- 그러나 위 순위들은 **전부 "전체 기간(글로벌)" 기준**이고, **장세로 쪼개서(conditioning) 순위를 내는 코드는 어디에도 없다.**
- `entry_regime` 라벨이 실제로 쓰이는 분석은 **딱 한 곳**(`system_auto_pilot.py:664`)뿐인데, 그조차 **모든 로직을 한 덩어리로 합산**해 "현재 국면의 켈리 베팅 사이즈"만 뽑는 용도다(수익 기반만, 방어·로직별 분해 없음).

> 즉, 당신이 원하는 **"장세 × 로직 교차표 + (수익·방어) 순위"** 는 현재 시스템에 **존재하지 않는다.** 핵심 근거: 코드 전체에 `groupby(["entry_regime", group_key])` 같은 **장세×로직 동시 집계가 단 한 건도 없다.**

## 1. 지금 "있는" 것

### 1.1 장세 판별 (Regime Detection) — ✅ 있음

| 파일 | 위치 | 하는 일 |
|------|------|---------|
| `regime_meta_analyzer.py` | `_classify_regime()` L156–224 | VIX·SPX·KOSPI로 `HIGH_VOL / BULL / BEAR / SIDEWAYS / UNKNOWN` 분류 (VIX>20→HIGH_VOL, 5일 레인지<3.5%→SIDEWAYS, MA20 상/하→BULL/BEAR) |
| `meta_governor.py` | `ACTION_BY_REGIME` L43–79, `_step_regime()` L1140–1171 | 권위 있는 `META_REGIME_KEY` 확정. 국면별 `kelly_cap`·S1/S4 비중 한도. 스트레스 우선순위 병합(`HIGH_VOL>BEAR>SIDEWAYS>BULL`) |

| `system_auto_pilot.py` | L644 | 텍스트 라벨에서 `BULL/BEAR/CHOP` 코어스 추출 |

→ "지금이 무슨 장세인가"는 잘 판단한다.

### 1.2 거래별 장세 라벨 저장 — ✅ 있음(쓰임은 빈약)

```744:744:forward/shared.py
    try: cursor.execute("ALTER TABLE forward_trades ADD COLUMN entry_regime TEXT DEFAULT 'UNKNOWN'")
```
- 진입 시 `entry_regime`에 당시 국면 기록 (`forward/shared.py:2265`, 컬럼 목록 `forward/ledger.py:73`).
- **그런데 이 라벨을 실제로 읽어 분석하는 코드는 사실상 1곳뿐:**

```663:671:system_auto_pilot.py
    # 1. 내 장부에서 '현재와 동일한 국면(Regime)'에 진입했던 청산 종목들만 추출
    regime_df = df[df['entry_regime'] == regime_key] if 'entry_regime' in df.columns else df

    if len(regime_df) >= 10: # 데이터가 충분할 때만 켈리 공식 가동
        r_wins = regime_df[regime_df['final_ret'] > 0]
        r_loses = regime_df[regime_df['final_ret'] <= 0]
        r_win_rate = len(r_wins) / len(regime_df)
        r_pf = r_wins['final_ret'].sum() / (abs(r_loses['final_ret'].sum()) + 0.1)
```
- **현재 국면만**, **모든 로직을 통째로 합산**, **수익(승률·PF)만** → `DYNAMIC_KELLY_RISK`(베팅 사이즈) 산출. **로직별 분해도, 방어 순위도, 상승/횡보/하락 3종 비교표도 아니다.**

| 엔진 | 파일·위치 | 그룹 키 | 장세 조건? | 지표 |

### 1.4 로직 순위 — 방어 기반 — ✅ 있음 (정교하지만 전부 글로벌)

데스매치 종합점수는 **수익과 방어를 한 점수로 합쳐 순위**까지 낸다:

```244:251:evolution/deathmatch_scorecard.py

    w_wr = float(weights.get("wr", 0.10))
    w_pf = float(weights.get("pf", 0.10))
    w_exp = float(weights.get("expectancy", 0.08))
    w_kelly = float(weights.get("kelly_sub", 0.05))

    w_mdd = float(weights.get("mdd_penalty", 0.32))   # ← 방어(최대낙폭) 페널티
    w_vol = float(weights.get("vol_penalty", 0.10))   # ← 방어(변동성) 페널티
```

```345:346:evolution/deathmatch_scorecard.py
        a.composite_score = float(composite)
```

| 방어 장치 | 파일·위치 | 장세 조건? | 성격 |
|-----------|-----------|-----------|------|
| 데스매치 종합점수(공격−MDD−변동성) | `evolution/deathmatch_scorecard.py` `compute_composite_v2` L236–355 | ❌ 글로벌 | **수익+방어 통합 순위** |
| 배틀로얄 하위 도태·폭락방어 면제 | `evolution/deathmatch_battle_royale.py` `run_battle_royal` L229–384 | ❌ 글로벌(룩백만) | 순위·도태 |
| Treasury 헬스 게이팅(승률·PF·연패·MDD→배수) | `meta_governor.py` `_build_treasury_health_and_mult` L492–549 | ❌ 글로벌* | 로직별 방어+수익 |
| 자본 데스매치(고정 vs 켈리, MDD·연패·방어금) | `capital_deathmatch.py` | ❌ 글로벌(`regime` 0건) | 자금관리 비교(로직별 아님) |

> *Treasury는 **윈도우 길이만** 국면에 따라 바뀐다(`resolve_asymmetric_treasury_lookback_days` L105–134: BEAR/HIGH_VOL 15~20일, BULL 120일). 하지만 **통계 자체를 국면 버킷으로 나누지는 않는다.** "최근 N일"일 뿐, "하락장에서의 성과"가 아니다.

`evolution/` 폴더 전체에서 `regime` 언급은 사실상 0건(ace_fact_pack 1건)으로, **모든 순위 엔진이 국면 비조건부**임을 재확인.

## 2. 지금 "없는" 것 — 당신 질문의 핵심 갭

| 원하는 기능 | 존재? |
|-------------|-------|
| 상승장에서 로직별 수익 순위 | ❌ 없음 |
| 횡보장에서 로직별 수익 순위 | ❌ 없음 |
| 하락장에서 로직별 방어(MDD/연패) 순위 | ❌ 없음 |
| 장세 × 로직 교차 성과표 (`groupby(entry_regime, group_key)`) | ❌ **코드 전무** |
| 장세별 "이 로직이 이 장세에 강하다/약하다" 판정 | ❌ 없음 |

**구조적 원인:** 순위를 매기는 모든 곳(데스매치/콜로세움/MAB/Treasury)이 거래를 **시장(KR/US) + 롤링 윈도우**로만 거르고 **`entry_regime`으로는 절대 거르지 않는다.** 그래서 "상승장 1등 로직 vs 하락장 1등 로직"을 비교할 산출물이 만들어지지 않는다.

```
[진입] 스캐너 → forward_trades 기록  (entry_regime 라벨 함께 저장됨 ✅)
                         │
                         ▼
[분석] ─ 데스매치/콜로세움/MAB ── group_key별 순위  ← 장세 무시(글로벌) ❌
       └ system_auto_pilot 엔진1.9 ─ 현재국면 전체합산 켈리  ← 로직분해·방어 없음 ⚠️
                         │
                         ▼

```

핵심: **`entry_regime` 라벨은 저장만 되고, "장세별 로직 순위"라는 형태로는 한 번도 소비되지 않는다.** 이미 켜둔 가로등(라벨)을 아무도 안 보고 있는 셈.

## 4. 만들려면 (권고 — 구현 시 별도 승인)

이미 **재료는 다 있다**(entry_regime 라벨 + 데스매치 수익·방어 점수 함수). 빠진 건 **장세로 쪼개는 groupby 한 겹**이다.

1. **장세×로직 교차 집계기 신설** — `forward_trades`를 `groupby(["entry_regime", group_key])`로 집계해 국면별 `n / win_rate / PF / mean_ret / MDD / 최대연패`를 산출.
2. **수익·방어 2축 점수 재활용** — `evolution/deathmatch_scorecard.compute_composite_v2`를 국면 버킷마다 호출(공격 z-score − MDD 0.32 − 변동성 0.10). → **국면별 종합 순위표**.
3. **3종 리더보드 출력** — `[상승장 TOP] / [횡보장 TOP] / [하락장 TOP]`를 각각 **수익 1위·방어 1위**로 분리 표기(텔레그램/리포트).
4. **실전 연결(선택)** — 현재 `META_REGIME_KEY`에 맞춰 "이 장세 강자 로직"에 켈리 가중, "이 장세 약자"에 비중 축소. 표본 부족 국면은 글로벌로 폴백(현 엔진1.9의 `>=10` 가드 패턴 재사용).
5. **표본 한계 주의** — 국면별로 나누면 표본이 급감(특히 HIGH_VOL/BEAR). 최소 표본 가드·베이지안 수축(EWC/Thompson) 필수.

## 5. 결론

- **장세 판별**과 **로직 순위(수익+방어)** 는 각각 잘 갖춰져 있으나, **둘이 교차된 "장세별 로직 순위"는 없다.**
- `entry_regime` 라벨이 이미 거래마다 박혀 있으므로, **데스매치 점수 함수에 `groupby(entry_regime, group_key)` 한 겹만 추가**하면 원하는 "상승/횡보/하락 × (수익·방어) 순위표"를 비교적 적은 작업으로 만들 수 있다.
- 현 상태의 가장 가까운 근사치는 `system_auto_pilot.py` 엔진 1.9뿐인데, 이는 **현재 국면·전체합산·수익전용 켈리**라서 "로직 순위"도 "방어 순위"도 "3장세 비교"도 아니다.

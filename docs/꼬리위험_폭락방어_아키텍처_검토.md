# 꼬리 위험(Fat-tail) · 폭락장 방어 아키텍처 코드레벨 검토

> 대상: 한국·미국 메인 퀀트(`bitget/` 제외)
> 관점: "단순 하드 브레이크로 인한 휩쏘(Whipsaw)" 우려에 대한 펀드매니저 실사
> 결론 요약(3줄):
> 1. 합성 블랙스완은 **일봉 단위 군집형 꼬리위험**은 잘 모방하나, **장중 Flash Crash(급락 후 당일 회복)** 는 구조적으로 모방하지 못한다.
> 2. 하드 브레이크(서킷 −5% / DEFCON ≤2)는 **신규 진입만 차단**하며 **기존 OPEN을 강제 청산하지 않는다.** → 휩쏘 위험은 낮으나, 서킷은 **자동 해제(OFF)가 없어 "고착(sticky) ON"** 이라는 반대 방향 위험이 있다.
> 3. `TAIL_RISK_FUND`의 VIX≥35 ×30 페이오프는 **레거시 `CENTRAL_TREASURY` 회계만 복원**하며, 실제 복리 자산인 **True NAV(`treasury_state.json`)에는 유입되지 않는다.** → 쿠션 효과가 포지션 사이징과 분리(decoupled)되어 있다.

## 1. 합성 데이터의 블랙스완 구조 — `synthetic_data_generator.py`

### 1.1 HMM 4국면 모수 (Merton 점프-확산)

`Regime` 데이터클래스는 일간 로그수익을 `mu(드리프트) + sigma·z(확산) + 점프`로 생성한다.

```49:65:synthetic_data_generator.py

    #         name          mu       sigma   lam    mu_jump  sig_jump vol_surge
    Regime("SIDEWAYS",   0.00000,  0.0110, 0.020,  0.000,   0.020,   1.00),
    Regime("BULL",       0.00120,  0.0130, 0.020,  0.006,   0.022,   1.15),
    Regime("BEAR",      -0.00110,  0.0210, 0.055, -0.012,   0.035,   1.40),
    Regime("BLACK_SWAN",-0.00450,  0.0480, 0.320, -0.040,   0.065,   3.20),
)
```

**BLACK_SWAN 모수 해부 (다른 국면 대비 배수):**

| 파라미터 | 의미 | SIDEWAYS | BEAR | **BLACK_SWAN** | BEAR 대비 |
|---|---|---|---|---|---|
| `mu` | 일간 로그 드리프트 | 0.0% | −0.11% | **−0.45%** | 약 4배 하락압 |
| `sigma` | 확산 변동성 | 1.1% | 2.1% | **4.8%** | 약 2.3배 |
| `lam` | 점프 발생률(Poisson/일) | 0.02 | 0.055 | **0.32** | 약 5.8배 (하루 점프 빈발) |
| `mu_jump` | 점프 평균(로그) | 0.0% | −1.2% | **−4.0%** | 약 3.3배 하방 점프 |
| `sigma_jump` | 점프 표준편차 | 2.0% | 3.5% | **6.5%** | 약 1.9배 |
| `vol_surge` | 거래량 서지 배수 | 1.0 | 1.4 | **3.2** | 약 2.3배 |

→ 블랙스완은 **음(−)의 드리프트 + 높은 확산 + 빈번하고 깊은 하방 점프**가 동시에 작동한다. `lam=0.32`는 블랙스완 하루당 평균 0.32회 점프 발생을 의미하고, 점프 1회 평균이 로그 −4%이므로 단일 일봉에서 −10%~−30%(클립 한계)급 붕괴가 실제로 발생한다.

**결론(질문 2):**
- −5% 서킷 / DEFCON ≤2 모두 **신규 진입 차단형(소프트)**, **기존 OPEN 강제청산 없음.**
- 따라서 보유 포지션 휩쏘(노이즈로 전량 손절 후 재매수)는 **구조적으로 발생하지 않음.**
- 단, **서킷의 자동 해제 부재**로 인한 "과방어 고착" 위험은 별도로 존재 → 권고안 §4 참조.

## 3. TAIL_RISK_FUND 쿠션 역할과 NAV 자금 흐름

### 3.1 적립과 ×30 페이오프 (System B 자율 조율, 주간)

- 적립: `TAIL_RISK_FUND ← min(CENTRAL_TREASURY × 1.5%, ...)` (목표 도달까지 국고에서 이전)
- 페이오프: `VIX ≥ 35` → `CENTRAL_TREASURY += TAIL_RISK_FUND × 30`, 펀드 0 리셋
- 손익구조: 평시 −1.5%(보험료)를 내고, 패닉장에 ×30(보험금)을 받는 **볼록(convex) 꼬리 헤지**. 국고의 1.5%로 국고의 45%(=1.5%×30)를 복원하는 비대칭 페이오프.

`live_nav_manager.py` 전체에 `TAIL_RISK_FUND`·`CENTRAL_TREASURY`·×30 페이오프를 참조하는 코드가 **하나도 없다.** 즉 ×30 페이오프는 `CENTRAL_TREASURY_{KR,US}`라는 **config(KV) 회계 항목**만 증가시키고, 포지션 사이징과 손실 상쇄의 실제 기준인 **True NAV(treasury_state.json)** 에는 반영되지 않는다.

이는 아키텍처 문서에도 명시된 의도적 분리다:

> `CENTRAL_TREASURY_{KR,US}`(초기 3억), `TAIL_RISK_FUND_{KR,US}`: **Tail Risk Convexity Shield** — 매주 treasury×0.015 적립, VIX≥35 시 ×30 페이오프. ※ M4에서 CENTRAL_TREASURY 직접회계는 deprecated(Live NAV 엔진이 100% 담당).
> — `docs/한미_퀀트_전체구조_100%_아키텍처.md` §6.3

### 3.4 자금 흐름도 (텍스트)

```
[평시]
  CENTRAL_TREASURY ──(주간 1.5% 적립)──▶ TAIL_RISK_FUND
                                              │
                          ┌───────────────────┴────────────────────┐
                          ▼                                          ▼
   [경로 A: 시뮬레이션 회계]                        [경로 B: 실물 헤지 — NAV에 실제 반영]
   VIX≥35 → ×30 payoff                        inverse_etf_sniper: 테일×30%캡으로
   → CENTRAL_TREASURY 복원                      인버스 ETF 매수 → 청산손익
   → (deprecated, True NAV 미반영) ⚠️             → forward_trades[INVERSE_ETF]
                                               → live_nav_manager.apply_realized_pnl
                                               → True NAV(treasury_state.json) 상쇄 ✅
```

**결론(질문 3):**
- VIX≥35 ×30 페이오프는 **볼록 꼬리 헤지의 "회계상 쿠션"** 으로, `CENTRAL_TREASURY` config 값을 복원한다. 그러나 이 항목은 M4에서 deprecated 되었고 **실제 복리 자산(True NAV)에는 유입되지 않는다.** → 보고서/대시보드 상의 쿠션과 실거래 NAV 쿠션이 **불일치**한다.
- NAV 손실을 *실제로* 상쇄하는 쿠션은 **인버스 ETF 스나이퍼**(테일 자금의 30% 한도로 실물 헤지 진입 → 청산손익이 NAV에 복리 반영)다.
